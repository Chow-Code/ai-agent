# SVN 策划文档同步与需求整理 Skill

## 工具概述

本 Skill 实现「策划文档从 SVN 获取 → 转为 Markdown → 同步到 docs/developdoc/策划原文」的流程。**仅当用户指定具体系统时**，才在同步后整理该系统的功能文档。

- **SVN 源**：`svn://172.16.4.12/TianLong3/Design/系统文档`
- **输出目录**：`docs/developdoc/策划原文/`
- **文档格式**：主要为 Excel (.xlsx) 和 HTML (.html)，转换为 Markdown (.md)

**⚠️ 职责边界**：
- **未指定系统**：仅同步策划原文到 `策划原文/`，**不**生成功能文档
- **指定系统**：同步策划原文 + 整理该系统的功能文档（`服务器功能点.md`、`配置表结构.md`、`变更记录.md`）

## 使用场景

- 需要将 SVN 上的策划文档同步到 Git 子仓库时
- 策划更新了 Excel 文档，需要同步到 developdoc 供团队查阅时
- **指定系统后**：同步策划原文并整理该系统的服务器功能点（一次完成）

## 用户输入与执行逻辑

| 用户输入示例 | 执行内容 |
|--------------|----------|
| `同步策划需求文档`、`更新策划案`、`同步策划文档` | **仅**同步策划原文到 `策划原文/`，**不**生成功能文档 |
| `同步策划需求文档 并整理 T-挑战副本` | 同步策划原文 + 整理 T-挑战副本 的功能文档 |
| `更新策划案 整理 Z-装备` | 同步策划原文 + 整理 Z-装备 的功能文档 |
| `更新策划案 整理所有`、`更新策划案 全量整理` | 同步策划原文 + 整理所有策划原文对应的功能文档 |

**判断规则**：用户**未指定具体系统**（如 T-挑战副本、Z-装备、S-商店）时，**只执行策划原文同步**；用户指定了系统名或「整理」「生成需求」+ 系统名时，才在同步后整理该系统的功能文档。

## 前置条件

- SVN 客户端已安装（`svn --version` 可用）
- Python 已安装（用于 markitdown）
- `uvx markitdown` 或 `python -m markitdown` 可用
- `docs/developdoc` 子模块已初始化（`git submodule update --init`）
- 有 SVN 和 Git 仓库的访问权限

## 自动化流程

```
步骤1：SVN export → markitdown 转换 → 复制到 docs/developdoc/策划原文 → git add/commit → push
步骤2（仅当用户指定系统时）：读取策划原文 → 生成/补充 docs/developdoc/{系统名}/ 下的 服务器功能点.md、配置表结构.md、变更记录.md
```

**说明**：未指定系统时只执行步骤1；指定系统时步骤1+2 一次完成。直接提交到 developdoc 的当前分支（develop），无需创建新分支或 PR。

## 命令参考

### 方式一：使用同步脚本（推荐）

```bash
# 完整同步（SVN export → 转换 → Git 提交）
cd .cursor/skills/svn-doc-sync/scripts
./sync.sh

# 预览模式（不实际执行）
./sync.sh --dry-run

# 仅同步指定文件
./sync.sh --only "A-暗器.xlsx"
```

**注意**：脚本仅完成同步；需求整理由 Agent 在同步完成后根据用户输入执行。

### 方式二：手动执行

```bash
# Step 1: SVN export 导出到临时目录
svn export --force "svn://172.16.4.12/TianLong3/Design/系统文档" /tmp/svn_export_design

# Step 2: 转换单个文件
uvx markitdown input.xlsx -o output.md

# Step 3: 复制到子模块并提交到当前分支
cp output.md docs/developdoc/策划原文/
cd docs/developdoc
git add .
git commit -m "sync: 策划文档同步"
git push origin HEAD
```

## 文件映射规则

- 保留原文件名前缀：`A-暗器.xlsx` → `策划原文/A-暗器.md`
- 子目录保留：`历史文档/Z-装备.xlsx` → `策划原文/历史文档/Z-装备.md`
- 支持格式：`.xlsx`、`.xls`、`.html`
- 输出目录：`docs/developdoc/策划原文/`（非 developdoc 根目录）

## 功能文档整理（仅当用户指定系统时执行）

### 输出范围

- **产出**：`docs/developdoc/{功能名}/服务器功能点.md`、`配置表结构.md`、`变更记录.md`
- **不产出**：`docs/design/server/` 下的任何文档

### 什么是「服务器关注的需求」

需要服务器参与**校验、计算、持久化、跨系统协作**的业务内容。详见同目录 `organize/服务器关注需求定义.md`。

### 执行流程

**提取策略**：结构识别 → 分类过滤 → 结构化输出。详见同目录 `organize/需求提取方法论.md`。

1. **读取策划原文**：`docs/developdoc/策划原文/{功能名}.md`
2. **重点扫描界面逻辑表格**：搜索 `ID | 界面单元 | 控件类型 | 备注说明` 结构，**备注说明列**常含服务器逻辑（次数、背包、奖励、销毁、领取等），必须逐行读取并纳入
3. **生成或补充完善** `{功能名}/服务器功能点.md`：按 `organize/服务器关注需求定义.md` 的 **F-N 模板**（含前置条件、编号主流程、关键约束/边界、失败分支、存储字段、推送、属性影响）；功能细节逻辑抽取见 `organize/需求提取方法论.md` **第六节**；若存在同主题 `docs/design/server/*服务器需求*.md` 可对齐粒度
4. **生成** `{功能名}/配置表结构.md`：从原文「数据配置结构」「数据配置」等 Sheet 提取
5. **生成或更新** `{功能名}/变更记录.md`：首次整理或增量更新

### 需求变更识别

**当用户指出「需求变化」「策划案变更」时**：
- 以用户提供的策划案/截图为准，**优先采纳用户描述**
- 若策划原文与用户描述不一致，需在变更记录中注明「策划案已更新，以用户提供为准」
- 界面单元（按钮/操作）变更时，需同步更新「操作 → 服务器逻辑」映射

### 快速检查需求变化（Git）

**用途**：快速定位策划文档的变更内容，明确哪些部分发生变化。

**命令示例**（`docs/developdoc` 为子模块时，需进入子模块目录执行）：

```bash
# 进入 developdoc 子模块
cd docs/developdoc

# 查看指定策划文档的提交历史
git log --oneline -- 策划原文/T-挑战副本.md

# 查看最近一次提交的详细 diff
git log -1 -p -- 策划原文/T-挑战副本.md

# 查看最近 N 次提交的 diff（如最近 3 次）
git log -3 -p -- 策划原文/T-挑战副本.md

# 与上一版本对比
git diff HEAD~1 -- 策划原文/T-挑战副本.md
```

**执行时机**：整理需求前、用户反馈「需求变化」时，可先执行上述命令查看策划原文的 Git 变更，再结合用户描述更新服务器功能点。

### 命令与触发

- **整理单个系统**：用户说「同步策划需求文档 并整理 T-挑战副本」或「整理 Z-装备」— 同步策划原文 + 整理该系统的功能文档
- **补充完善**：用户说「补充完善 S-商店」— 对照策划原文核对已有服务器功能点，补充遗漏
- **整理全部**：用户说「更新策划案 整理所有」— 同步策划原文 + 对根目录下所有 .md 执行功能文档整理（排除 `历史文档/`、`待审核/`）

### 注意事项

- `历史文档/` 和 `待审核/` 子目录下的文件**不需要整理**，永久排除
- 增量更新时，**合并**到现有需求并追加变更记录，不重写全文
- 若用户**未指定具体系统**，**不执行**功能文档整理步骤，仅同步策划原文
- **界面逻辑表格**：凡含「界面单元」「备注说明」的表格必须逐行读取，备注中出现次数/背包/奖励/销毁/领取等词即纳入服务器功能点

### 常见漏检自检（需求识别失败时）

执行需求整理后，若用户反馈「需求识别错误」「策划案已变更」，请对照 `organize/需求提取方法论.md` 中的「常见漏检清单」与 **第六节（功能细节逻辑提取模式）** 自检；并对照 `organize/服务器关注需求定义.md` 文末检查清单（含 F-N 编号、约束/边界、属性影响、推送）。以用户提供的策划案为准优先更新。

## 增量同步策略

脚本支持：
- `--dry-run`：预览将要转换的文件，不实际执行
- `--only <文件名>`：只转换指定文件

## 子目录处理

SVN 目录包含：
- 根目录：约 30 个 .xlsx/.html 文件
- `历史文档/`：历史版本 Excel
- `待审核/`：待审核文档

脚本会递归处理所有子目录。

## 注意事项

- markitdown 对 Excel 的转换会将每个 Sheet 转为一个 Markdown 表格，复杂合并单元格可能丢失格式
- 中文文件名在 Windows 下需注意编码（GBK/UTF-8）
- 转换失败的文件会跳过并记录日志，不中断整体流程
- **一次执行**：同步策划原文 + 功能文档整理（若用户指定系统）在同一轮对话中完成，避免分步导致生成失败
- **默认仅同步**：用户说「同步策划需求文档」且未指定系统时，只同步策划原文，不生成功能文档
