# 服务器关注需求定义

> 基于 `docs/design/server` 中已整理的服务器需求文档总结，供 svn-doc-sync Skill 需求整理时参考。

## 文档层级关系

| 层级 | 路径 | 用途 | 产出方 |
|------|------|------|--------|
| **策划原文** | `docs/developdoc/策划原文/{功能名}.md` | 策划原始需求（markitdown 产物） | svn-doc-sync |
| **精简需求** | `docs/developdoc/{功能名}/` | 服务器侧业务逻辑提炼，快速评估工时 | **svn-doc-sync（需求整理步骤）** |
| **开发需求** | `docs/design/server/{功能名}/` | 面向实现的完整需求（DDD、协议、存储、流程图） | 下游人工/工具，**非本 Skill** |

**⚠️ 需求整理仅产出**：`docs/developdoc/{功能名}/服务器功能点.md`、`配置表结构.md`、`变更记录.md`。**不**产出 `docs/design/server/` 下的任何文档。

## 什么是「服务器关注的需求」

**定义**：需要服务器参与**校验、计算、持久化、跨系统协作**的业务内容。满足以下任一条件即视为服务器关注：

### 1. 数据校验

- 条件判断（开放条件、关闭条件、等级限制、职业限制）
- 权限/状态校验（是否可购买、是否可穿戴、是否已激活）
- 次数/上限校验（限购次数、剩余可购买次数）

**示例**：商店开放判断、商品开放判断、装备穿戴职业/部位/等级校验

### 2. 数据持久化

- 需要存储到 Redis（玩家数据、角色数据、进度数据）
- 需要记录状态（购买次数、强化等级、失败次数、星孔激活）
- 需要持久化配置引用（如生效配置 Id）

**示例**：roleId + 商品ID 购买次数；roleId + slotType 强化等级；playerId 副本进度

### 3. 数据计算

- 需要服务器计算的公式（成功率、加成、阈值）
- 算法逻辑（匹配值、段位、排行榜）
- 概率/随机（掉落、强化成功判定）

**示例**：强化成功率 = 基础 + 失败次数加成；经验加成算法

### 4. 跨系统协作

- 调用其他系统（扣货币、发道具、背包、掉落、邮件）
- 依赖其他子域能力（背包消耗、装备穿戴、属性计算）
- 事件驱动（怪物死亡、阶段完成、结算）

**示例**：购买扣货币 → 发道具；装备穿戴卸下 → 背包转移；副本结算 → 发放奖励

### 5. 配置驱动

- 依赖配置表字段（表名:字段名）
- 配置解析（条件表、属性表、掉落表）
- 配置变化驱动业务（版本变更、周期刷新）

**示例**：Exchange:CoinType、Cost、LimitType；Shop:OpenCondition、CloseCondition

### 6. 协议/接口

- 客户端请求/响应（请求参数、响应字段、错误码）
- 服务器推送（推送时机、推送内容）
- 跨服务协议（战斗服回调、中心服转发）

**示例**：MsgCtrReqEquipWear / MsgCtrResEquipWear；推送 MsgCtrNtfRoleEquipmentChange

### 7. 业务规则

- 限购规则（日/周/月限购）
- 刷新规则（商店关闭刷新、周期 0 点刷新）
- 顺序规则（先扣货币再发道具）

**示例**：商店关闭时刷新该商店内所有道具限购；每日 0 点刷新日限购商品

### 8. 防刷/安全

- 防刷逻辑（单日大量购买报警、截停）
- 数据一致性（事务、幂等）

**示例**：单一玩家一日内大量购买某种道具需后台报警，超过规模需截停

### 9. 界面交互逻辑（含服务器侧行为）

**定义**：界面按钮/操作背后的业务规则，需服务器参与。策划常以「界面单元表格」描述：`ID | 界面单元 | 控件类型 | 备注说明`。

**备注说明中的服务器逻辑**（满足任一即纳入）：
- 次数变化（如「挑战次数-1」「次数不做变化」）
- 临时背包/奖励处理（如「销毁临时背包奖励」「领取当前临时背包奖励」）
- 数据持久化（如「发放至正式背包」「扣除道具」）
- 状态变更（如「结算后响应离开副本事件」）

**示例**：挑战失败界面「放弃奖励」= 销毁临时背包、次数不变；「领取奖励」= 发放临时背包、次数-1；「离开副本」= 回到主界面（需服务器响应）

**识别规则**：凡表格含「界面单元」「备注说明」列，且备注说明中出现次数、背包、奖励、销毁、领取、扣除等词，**必须**纳入服务器需求。

## 非服务器关注（必须过滤）

| 类型 | 说明 | 示例 |
|------|------|------|
| 纯客户端 UI | 界面布局、按钮样式、图片引用 | 元素位置、=DISPIMG 引用 |
| 美术需求 | 模型、特效、贴图、动画 | 3D 模型、特效 ID |
| 参考分析 | 竞品分析、设计思路 | 参考游戏分析 |
| 表现逻辑 | 仅客户端展示、无服务器校验 | 纯展示用的倒计时 |

**注意**：界面设计逻辑中**引用配置字段**的部分（如按钮是否可点击依赖配置）属于服务器关注，需纳入。

## 与 docs/design/server 的对应关系

`docs/design/server` 下的服务器需求文档是**完整开发需求**，包含：

| 章节 | 内容 | 与精简需求的关系 |
|------|------|------------------|
| SR-1 业务概述 | 业务背景、业务价值、影响范围 | 精简需求「系统概述」的扩展 |
| SR-2 DDD 架构设计 | 领域、设计模式、新增/调整文件 | 基于精简需求「业务逻辑」做架构设计 |
| SR-3 协议设计 | 请求/响应/推送、消息结构 | 基于精简需求「主流程」设计协议 |
| SR-3.1 业务流程图 | Mermaid 时序图、流程图 | 基于精简需求「主流程」绘制 |
| SR-4 数据存储设计 | Redis Key、数据结构、存储策略 | 基于精简需求「存储维度」设计 |

**提炼原则**：精简需求应足够支撑下游编写 `docs/design/server` 下的 `*服务器需求*.md`，除「做什么」「主流程」「关键配置」「存储」外，须尽量补齐 **前置条件、关键约束/边界、失败分支、推送、属性影响**（有则填）；**功能细节逻辑**的抽取规则见 `需求提取方法论.md` **第六节**。

## 精简需求输出字段要求（`服务器功能点.md`）

每个**服务器关注**的能力点使用 **`### F-N {功能点名称}`** 编号（N 从 1 递增，与策划功能清单或模块顺序对齐，便于变更记录与评审引用），字段如下：

```markdown
### F-N {功能点名称}
- **做什么**：一句话概括服务端职责（校验谁、算什么、写哪里、与谁协作）。
- **前置条件**：触发本功能前须满足的状态/参数（等级、开关、次数、阵容等）；无则写「无」。
- **主流程**：
  1. …（校验 / 读配置）
  2. …（扣费、扣道具、计算）
  3. …（改内存、写 Redis、发奖、发邮件等）
- **关键约束/边界**：操作顺序（先…再…）、失败是否回滚、互斥/幂等、与其他子域的先后或权威约定。
- **失败分支**：`{校验点} → {失败条件} → {结果或错误语义}`；策划已给错误码符号名的照录，否则写「待定」，禁止编造。
- **存储字段**：`{Key 模式或维度说明}` + 须持久化的**字段名级**要点（若策划未给字段名，写清「须记录的状态语义」）。
- **推送**：推送时机 + 内容语义；消息名未定时可写「待设计」，但须标明「需推送 / 仅响应内回包 / 客户端轮询即可」等。
- **属性影响**（有则填，无则省略本行）：生效范围（仅角色 / 单宠 / 全员 GlobalTree 等）；是否配置终值行（扣旧加新）；登录是否重建相关属性。
- **关键配置**：反引号列出 `表名` 或 `表名.字段名`（与策划表一致）；无表则写「无」。
```

**与 `docs/design/server` 的对齐**：整理时若已存在同主题的 `*服务器需求*.md`，可对照其 **SR-1 子节、功能清单、步骤与错误码** 补全上述字段，使 `developdoc` 与正式需求**语义同级**（仍不复制 SR-2 DDD 等实现章节）。

## 检查清单（生成时自检）

- [ ] 每个功能点是否满足「服务器关注」定义（校验/持久化/计算/协作/配置/协议/规则/防刷/界面交互逻辑）
- [ ] 是否过滤了纯客户端/美术/参考分析
- [ ] **界面逻辑表格**：是否扫描了「界面单元」「备注说明」列，备注中的次数/背包/奖励/销毁/领取等逻辑是否已纳入
- [ ] 主流程是否为**编号步骤**且包含关键校验
- [ ] 关键配置是否列出 `表名` / `表名.字段名`（反引号）
- [ ] 存储是否达到**字段名级**或明确「须记录的状态语义」（便于评估 Redis Key 与 Storage）
- [ ] 红点规则是否提炼了触发条件（服务器需判断的）
- [ ] 每个功能点是否带 **`F-N` 编号**（从 F-1 起，与变更记录可交叉引用）
- [ ] **关键约束/边界**是否写清（尤其操作顺序、失败是否回滚/不写存储）
- [ ] **属性影响**：凡涉属性加成/战力/树变更的条目是否已写**生效范围**（角色/单宠/全员）及终值/登录语义（无则省略）
- [ ] **推送**：凡写存储或跨端可见状态变更的条目，是否已判定需推送或注明「仅响应内回包/轮询」
- [ ] **跨域权威**：登录时间、阵容、属性、阵位等易争议字段是否点明**唯一权威 Storage**（避免下游重复争论）
- [ ] **协议占位**：复杂能力是否已给出 `MsgCtrReq/Res/Ntf*` 占位（即使消息名「待设计」）便于下游直接接入
- [ ] **粒度对齐**：是否参照 `docs/developdoc/C-宠物/C-宠物-服务器功能点.md` 的表格结构（**逻辑点 / 服务器职责 / 主流程 / 存储维度 / 失败提示** + 关键规则/配置脚注）
