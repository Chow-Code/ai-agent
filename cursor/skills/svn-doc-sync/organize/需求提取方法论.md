# 需求提取方法论

> 基于市面需求提取工具（ReXCL、LangExtract、ReqInOne、Firecrawl 等）的实践总结，供 svn-doc-sync Skill 需求整理时参考。

## 一、行业工具概览

| 工具 | 定位 | 核心能力 | 可借鉴点 |
|------|------|----------|----------|
| **ReXCL** | 需求工程专用 | PDF/DOC→Markdown 转换；需求分类（功能/非功能） | 章节标题识别、表格结构保留、分类思维 |
| **ReqInOne** | LLM 需求规格生成 | 模块化：摘要→提取→分类，每步独立 prompt | 分步执行、降低幻觉 |
| **LangExtract** | 通用信息抽取 | Schema 约束、源定位、长文档分块 | 结构化输出、可追溯 |
| **Firecrawl** | 网页/文档抽取 | prompt + JSON schema 双模式 | 灵活与严格兼顾 |

## 二、本 Skill 采用的提取策略

### 1. 模块化分解（借鉴 ReqInOne）

将「从策划原文到精简需求」拆成**三步**，每步聚焦单一目标：

| 步骤 | 目标 | 输出 |
|------|------|------|
| **1. 结构识别** | 识别策划原文的 Sheet/章节结构（系统概述、界面逻辑、服务器逻辑、数据配置） | 确定提取来源 |
| **2. 分类过滤** | 按「服务器关注」定义判断每块内容是否纳入 | 功能点清单、配置表清单 |
| **3. 结构化输出** | 按固定模板填充，保证输出格式一致 | 服务器需求.md、配置表结构.md |

**执行时**：Agent 可先通读原文完成步骤 1，再逐块执行 2、3，避免一次性处理导致遗漏。

### 2. 结构化输出约束（借鉴 LangExtract）

**固定输出结构**：每个功能点按 **F-N 编号模板**逐项填充（见 `organize/服务器关注需求定义.md`「精简需求输出字段要求」与 svn-doc-sync `SKILL.md` 输出模板），除「做什么」「主流程」「关键配置」「存储」外，须覆盖**前置条件、关键约束/边界、失败分支、推送、属性影响**（有则填）；**功能细节逻辑**的提取规则见本文 **第六节**。

**好处**：
- 减少幻觉：模型有明确「填空」目标
- 便于下游：开发文档可直接引用字段
- 可追溯：缺失项易被发现

**实现**：使用输出模板（见 SKILL.md），Agent 按模板逐项填充。

### 3. 来源可追溯（借鉴 LangExtract 源定位）

**策划原文结构**：markitdown 产物通常含多个 Sheet（如「3.服务器逻辑」「数据配置」），每个功能点应能对应到原文位置。

**实践**：
- 服务器需求.md 顶部注明「来源：docs/developdoc/策划原文/{功能名}.md」
- 补充完善时，对照原文逐项核对，变更记录中写清「影响的功能点」
- 配置表结构.md 与策划原文「数据配置结构」「数据配置」Sheet 一一对应

### 4. Markdown 结构保留（借鉴 ReXCL）

**策划原文特点**：markitdown 从 Excel 转换，含表格（`|` 分隔）、NaN、Unnamed 列。

**处理规则**：
- **保留**：表格的列结构、有效字段名、类型、说明
- **清理**：NaN、Unnamed、空行
- **识别**：章节标题（如「3.2 道具购买逻辑」）对应功能点；表名（如「商品表（Exchange）」）对应配置表

### 4.1 界面逻辑表格识别（重点防漏）

**表格结构**：策划常用 `ID | 界面单元 | 控件类型 | 备注说明` 描述界面按钮/操作。

**提取规则**：
1. **定位**：搜索「界面单元」「备注说明」列，或「XX界面」标题下的表格
2. **备注说明必读**：该列常含服务器逻辑，如「点击后销毁临时背包奖励，挑战次数不做变化」
3. **触发词**：备注中出现「次数」「-1」「不变」「销毁」「领取」「临时背包」「发放」「扣除」等时，**必须**纳入服务器需求
4. **按界面分组**：同一界面（如「挑战失败界面」「挑战成功界面」）下的多个表格行，合并为一条功能点，主流程为「操作 → 服务器逻辑」映射

### 5. 分类思维（借鉴 ReXCL）

**ReXCL 分类**：Info、Header、Functional Requirement、Non-Functional Requirement。

**本 Skill 分类**：
- **纳入**：满足「服务器关注」定义（校验、持久化、计算、协作、配置、规则、防刷）
- **过滤**：纯客户端 UI、美术、参考分析、仅展示逻辑
- **边界**：界面设计逻辑中**引用配置字段**的纳入（如按钮可点击依赖配置）

## 三、提取时的执行顺序建议

1. **（可选）查看 Git 变更**：`cd docs/developdoc && git log -3 -p -- 策划原文/{功能名}.md`，快速定位策划文档近期变更，便于聚焦变化部分
2. **通读原文**：识别「系统概述」「服务器逻辑」「数据配置」等 Sheet 边界
3. **先提取配置表**：从「数据配置结构」「数据配置」Sheet 提取表结构，建立字段索引
4. **再提取业务逻辑**：从「服务器逻辑」「系统规则&流程」等 Sheet 提取功能点，引用已建立的配置表字段
5. **重点扫描界面逻辑表格**：搜索「界面单元」「备注说明」列，逐行检查备注中的服务器逻辑（次数、背包、奖励、销毁、领取等）
6. **最后核对**：用常见遗漏检查清单与 **第六节** 五类规则逐项核对，补充失败分支、约束、推送、属性影响等
7. **（推荐）对照设计文档粒度**：若 `docs/design/server/{同功能或近似功能}/` 下已有 `*服务器需求*.md`，提取时对照其 **SR-1 子节、功能清单表、步骤编号、错误码、存储字段** 的写法，使 `developdoc` 精简需求尽量接近下游可核对的粒度（本 Skill 仍**不**产出 `docs/design/server` 文件）

## 四、常见漏检清单（需求识别失败时自检）

| 漏检类型 | 说明 | 应对措施 |
|----------|------|----------|
| **界面单元变更** | 策划案中界面按钮/操作已变更（如「再次尝试」→「放弃奖励」），原文或需求未同步 | 以用户提供的策划案/截图为准；若用户指出「需求变化」，优先采纳用户描述并更新需求 |
| **备注说明忽略** | 界面表格的「备注说明」列含服务器逻辑，但被当作纯 UI 过滤 | 凡含「界面单元」「备注说明」的表格必须逐行读取备注，出现次数/背包/奖励等词即纳入 |
| **临时数据流** | 临时背包、阶段掉落、结算前状态等中间数据流未提炼 | 备注中出现「临时」「阶段」「结算」时，提炼存储维度与生命周期 |
| **次数扣减时机** | 次数何时扣减（进入时/领取时/放弃时）未明确 | 备注中「次数」「-1」「不变」等必须转化为「操作 → 次数变化」的明确规则 |

## 六、功能细节逻辑提取模式

> 对齐 `docs/design/server` 中成熟服务器需求文档的常见写法（如 **功能清单 F-N**、**编号主流程**、**顺序与失败回滚**、**字段级存储**、**错误码/推送**、**属性生效范围**），在从策划原文提炼 `developdoc/{功能名}/服务器功能点.md` 时主动抽取下列信息，便于服务端快速理解与核对逻辑。

### 1. 步骤顺序提取规则

- 凡出现「先…再…」「之后」「若…则…」「否则」「失败时」等**顺序或条件**表述，必须拆为 **编号主流程**（1. 2. 3.），每步标明是校验、读配置、扣费/扣道具、改内存、写存储、发奖等。
- **失败与存储**：若策划写明「校验失败不扣费」「Ret≠0 不改状态」等，在**关键约束/边界**或**失败分支**中写清，避免默认「已改存储」。

### 2. 关键约束/边界提取规则

- **操作顺序**：如「先校验目标再结算旧状态再写入新状态」；禁止仅合并为一句「切换地图」。
- **互斥与幂等**：多方案/多入口是否互斥；重复请求是否允许、是否需幂等。
- **跨子域边界**：与背包、邮件、属性、战斗等协作时，写明**谁权威、谁先谁后**（若策划有写则必录）。

### 3. 属性逻辑提取规则

- **生效范围**：仅当前角色 / 单宠 / **全员（账号维度 GlobalTree）** 等；与 `docs/design/server` 及属性 Skill 语义一致，禁止把「全员」写成「给每个角色各加一份到角色树」。
- **数值形态**：配置是否 **按行终值**（升级/升档需 **扣旧加新**）；登录是否 **重建属性树**（非从 Redis 读整棵树）。
- **深度实现**：编写或评审精简需求中的属性句时，可对照 `.cursor/skills/attribute-system/SKILL.md`。

### 4. 错误分支提取规则

- 将策划或界面备注中的校验，落实为 **「校验点 → 失败条件 → 结果」**；若策划/原文已给出 **错误码符号名**（如 `proto` 中枚举名），**照录**；未给出则写 **「待定」**，**禁止编造**错误码。

### 5. 推送提取规则

- 凡「通知客户端」「刷新界面」「红点」「同步状态」等，提炼 **推送时机**（随请求回包 / 独立推送 / 登录全量等）与 **内容语义**（哪些数据变化）。
- 若策划未给消息名，可写 **「待设计」**，但在精简需求中仍应有一句 **「需推送」或「客户端轮询/读表即可」** 的判定，避免遗漏协作约定。

## 七、上下游对照规律与写法启示

> 通过对比 `developdoc/{功能}/服务器功能点.md`（精简需求）与 `docs/design/server/{功能}/*服务器需求*.md`（开发需求）发现：下游文档普遍补齐**编号、协议、字段级存储、错误码、推送时机、流程图、跨域权威**等内容。**精简需求若提前覆盖以下要点，下游加工成本最低**（C-宠物为本仓库当前**最佳示范**）。

### 1. 下游普遍补、上游普遍缺（共性规律）

| 维度 | 上游常见写法 | 下游普遍升级 |
|------|--------------|--------------|
| 章节 | bullet 列表 | **`SR-N` 强制章节**（业务/DDD/协议/存储/流程图） |
| 编号 | 无 | **`F-1～F-N` 功能清单** + SR-1.x 子规则 + 主流程 1./2./3. |
| 协议 | 零散或缺 | 统一 **`MsgCtrReq/Res/Ntf*`** + proto 字段表 |
| 存储 | 「持久化」一句话 | **`MsgData*Storage` + 字段名 + 兼容语义** |
| 错误 | 自然语言 / `Tip` ID | **`EMsgErrorType` 常量名 + 触发条件** |
| 推送 | 「成功通知」 | **触发场景表 + 通用 Ntf 路径**（如 `source="store"`） |
| 跨域 | 隐式「服务端为准」 | **唯一权威 Storage、登录写点、禁止双写** |
| 流程 | 短句 | **Mermaid sequenceDiagram + Handler/AppService 分层 Note** |
| 属性 | 「全员生效」 | **代码可对齐命名**：`BaseAttrAllRoles`、`GlobalTree` 等 |

### 2. 写精简需求时的 8 条收紧建议

1. **F-N 占位先行**：每个能力先给 `F-N` + `MsgCtrReq/Res/Ntf` 占位（无名时写「待设计」），减少下游重新编号成本。
2. **顺序敏感必拆步骤**：「多入口同一逻辑」「先 X 再 Y」一律编号步骤 + 标注「失败是否回滚/是否改存储」。
3. **存储三件套**：归属子域 + 键维度（`roleId` / `playerId` / `petId` 等）+ **字段名级语义**；避免只写「持久化」。
4. **错误三件套**：业务失败类型 + 是否 Ret≠0 + 是否走配置 ID（如 `Tip`）；策划无错误码就写「待定」，**禁止编造**。
5. **跨域读写权威**：登录时间、离线时间、阵容、属性、阵位等易争议字段，**精简需求里就点明唯一权威 Storage**。
6. **属性语义上抬**：用「全员 / 仅装备者 / 仅当前角色 / 仅单宠」等**与 attribute-system 三棵树直接对应**的措辞，便于下游直接落到 `BaseAttrAllRoles`、`GlobalTree` 等命名（详见 `.cursor/skills/attribute-system/SKILL.md`）。
7. **推送判定不可省**：每个**写存储**或**跨端可见状态变更**的能力，必须三选一标注「需独立推送 / 仅响应内回包 / 客户端轮询/读表即可」。
8. **以 C-宠物 为精简需求示范**：参照 `docs/developdoc/C-宠物/C-宠物-服务器功能点.md` 的表格粒度（**逻辑点 / 服务器职责 / 主流程 / 存储维度 / 失败提示** + 关键规则/关键配置脚注）；新功能整理时优先复刻该结构。

### 3. 典型转化举例（上游一句话 → 下游展开）

- **G-挂机·切换地图**：上游「先校验目标可进入再结算」 → 下游「校验目标 `AfkDungeonId` → 失败 Ret≠0 不改 `Pending`/锚点 → 旧图按 `LastDropTickUnix` 推进 → 写入新 `CurrentAfkDungeonId/CurrentBossLevel` → 推送 `MapChange`」。
- **S-商店·购买**：上游「先校验 → 再扣 → 再发奖」 → 下游 Mermaid：`MsgCtrReqStoreBuy` → 限购校验 → `BackpackAppService.AddItems` → `MsgCtrNtfItemChange(source="store")`，并标注**元宝补足绑元**。
- **J-绝学·属性生效**：上游「招式：基础属性对全员生效；技能效果对使用者生效」 → 下游 `BaseAttrAllRoles` / `BaseAttrEquippedRole`（直接对接代码命名与属性树语义）。
