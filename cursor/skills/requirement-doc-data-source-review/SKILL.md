# 需求文档与数据来源 Review

## 本技能解决什么

对照 **《服务器需求》与策划/需求澄清**，评审设计类文档是否**内容完整、不自相矛盾**，并**验证配置数据来源**是否能在仓库里落地；同时从**策划需求逻辑**上判断**是否可行、是否缺项需澄清**，减少定稿后才发现漏洞。

1. **需求覆盖**：文档中的功能点、参数、边界是否与 SR / 已确认澄清一致，有无遗漏或未闭环项。
2. **策划逻辑与可行性**：玩法/业务流程是否**闭环**；前置条件、分支、边界与异常是否说清；规则之间是否**自相矛盾**；是否存在「未定义就无法执行」的步骤（本层只审**文档中的逻辑**，不审代码实现）。
3. **数据来源**：文中提到的表、字段、`PARAM_*`、`Ct*`、`excel/csv/...` 是否**有出处**；是否缺「该表尚未建」等说明。
4. **CSV 可验证**：对应 **`excel/csv/{表}.csv` 是否存在**；文档写的**列名**是否在表头出现；若文档依赖某行/某 Id，**数据行是否存在**（打开 CSV 或 grep 核对）。
5. **澄清队列（内部）**：显式列出**待策划/程序/产品确认**的问题，**合并为有序队列**；**对用户输出时**必须遵守下文「**澄清交互协议**」，**禁止**一次甩出整张清单让用户自己消化。

**明确不做**：代码实现评审、DDD/架构评审、Redis/数据库表结构设计评审、Handler 与生成代码审查。

## 澄清产出物落盘（强制）

评审或逐条澄清后**需要落盘**时，遵守以下约定，避免「澄清」与「设计文档」两套真源：

1. **禁止**默认新建独立澄清类文件（如 `需求澄清.md`、`xxx-澄清.md`、`clarification.md` 与主文档平行），**除非用户明确要求**单独交付一份澄清记录。
2. **默认**将澄清结论写入**既有**设计文档中的专章，例如：
   - **`docs/design/server/{系统}/{系统}-开发文档.md`** 内增加或更新 **「需求澄清与评审结论」** 小节（推荐编号为 **DD-1.2** 或项目约定章节，与 `development.md` 模板兼容），包含：评审发现、澄清结论、仍须产品确认项、修订记录。
3. **同步**在 `docs/developdoc/{功能名}/服务器需求.md` 与 `**-服务器需求.md` 中**引用**上述专章（或保留摘要表），避免重复粘贴大段文字时与开发文档冲突。
4. **单一事实来源**：实现与测试以 **开发文档 §澄清专章** 与 **SR** 一致表述为准；若 SR 与开发文档冲突，以**已评审定稿**的为准并修订另一份。

执行本技能时，若用户说「不要新增文档」「澄清写进开发文档」，**直接**在 `{系统}-开发文档.md` 中维护专章，不新建平行 Markdown。

**与独立《需求澄清文档》并存**：若领域下已存在 **`docs/design/server/{系统}/{系统}-需求澄清文档.md`**（例如通过 **planning-requirement-clarification-doc** 技能从策划原文生成），评审时**优先**将其作为策划对齐与澄清条目真源；`{系统}-开发文档.md` 的 **DD-1.2** 以**摘要 + 链接**引用该文件即可，避免与策划原文顺序两套维护。仍须逐条澄清时交互协议不变。

## 何时使用

- 定稿或迭代 **服务器需求 / 开发文档 / 配置使用** 前
- 核对「策划案里有的，文档里写全没有」「配表缺哪张」
- 验证「文档里引用的 CSV 列在表里有没有、关键配置行在不在」
- 评审「策划案这条线能不能跑通、有没有没说清楚的」

## 输入与范围

| 优先级 | 文档 | 说明 |
|--------|------|------|
| 锚点 | `docs/design/server/{领域}/**-服务器需求.md` | 需求范围、已确认结论以 SR 为准 |
| 可选锚点 | `**-需求澄清文档.md` | 若存在，与策划原文对齐的澄清骨架/结论优先于此；与开发文档 DD-1.2 交叉引用 |
| 常评 | `**-开发文档.md`、`**-配置使用.md` | 参数表、表字段说明与 SR 交叉引用 |
| 事实来源 | `excel/csv/*.csv` | 配表是否已有文件、列与数据行 |
| 可选 | `excel/code/Ct*.go` | 列名与生成结构体字段一致时作辅助核对 |

若某块写「详见服务器需求」，以 SR 对应章节为需求依据，不在此技能中评「实现好坏」。

## 评审流程（建议顺序）

1. **列需求清单**：从 SR 抽出必交付的配置项、边界、待确认项（若 SR 无专章，则按用户给定的需求范围）。
2. **对照被评文档**：开发文档 / 配置使用是否**覆盖**上述条目；缺项 → 记入「缺口」。
3. **提取所有数据来源引用**：`excel/csv/` 路径、`Ct*`、表名、列名、`PARAM_*`。
4. **在仓库验证 CSV**（须实际查文件，不可凭记忆断言）：
   - **文件存在**：`excel/csv/{Table}.csv` 路径存在。
   - **列存在**：文档中的字段名与 CSV **第 1 行列名**（或与项目约定的表头行）一致。
   - **数据存在（按需）**：若文档写「默认用 Id=x」「引用第 y 行」或业务强依赖某条配置，在数据区检索该 Id/关键字，确认**至少一行**存在。
5. **策划逻辑走查**（在 SR/开发文档中按用户旅程或主流程逐段追问）：入口条件、成功/失败分支、互斥规则、与其它系统接口是否交代；发现**断点、循环依赖、未定义术语** → 记入「待澄清」。
6. **澄清项汇总（内部）**：文档已有 `Q-xxx`、待策划、待确认 + 本流程新发现的缺口，**合并为有序队列**（编号 C-01…C-N），**不**在首轮一次性全部展示给用户。
7. **对用户的输出**：先给**非澄清类**评审摘要（需求对照、CSV 验证、文档层级/输入质量等）；若有 N>0 条待澄清，**只展示第 1 条 + 选项**，并说明「共 N 条，答完本条后继续下一条」。后续轮次**每次仅一条 + 选项**，直到队列耗尽 → 再给出最终结论与可定稿判断。
8. **输出结构**：见下文「首轮模板」与「单条澄清模板」。

## 维度一：与需求一致、无遗漏

| 检查项 | 说明 |
|--------|------|
| 与 SR 对齐 | 功能、参数、边界、错误语义与 SR 是否一致；冲突或 SR 已更新而文档未改 → 修订项 |
| 缺口 | SR 要求写的配置/表/参数，文档是否都出现 |
| 未闭环 | 待确认项是否仍阻塞「定稿」 |

## 维度二：策划需求逻辑与可行性

**目标**：在**需求/策划表述**层面判断规则能否自洽执行，并标出必须澄清的空洞（不评价程序怎么写）。

| 检查项 | 说明 |
|--------|------|
| 主流程闭环 | 从触发到结束是否步骤完整；中间状态是否都有定义 |
| 前置与权限 | 谁能在什么条件下操作；不满足时行为是否写明 |
| 分支与边界 | 上限/下限、空列表、0、满、过期等是否覆盖或明确「另案」 |
| 规则一致性 | 不同段落对同一概念（类型、优先级、互斥）是否冲突 |
| 与外部衔接 | 依赖其它系统/协议/配表时，是否交代输入输出或引用位置 |
| 不可执行点 | 是否存在「若 A 则 B」但 A/B 的判定条件未定义 |

**危险信号**：仅有一句业务目标而无操作步骤；多处「待定」「后续补充」却无 `Q-xxx` 跟踪；两个规则隐含矛盾。

## 维度三：数据来源与 CSV 验证

| 检查项 | 说明 |
|--------|------|
| 路径与表 | 文档写的 `excel/csv/Xxx.csv` 在仓库中**存在**；或正文明确「待新增」且有计划 |
| 列名 | 文档中的 `CtXxx.Field` 与 CSV 列名（及生成代码字段名）**一致** |
| 枚举/类型 | 需求若约定某列为枚举，文档是否与 SR/策划一致（本技能不审 proto 实现，只审**文档表述是否自洽**） |
| 策划 Markdown | `docs/developdoc/**/配置表结构.md` 等**不能**替代「本仓库已有 `excel/csv`」；缺 CSV 即标缺口 |
| 数据行 | 对强依赖的默认行、模板 Id 等，在 CSV **数据区**检索确认存在 |

**操作提示**：用 **Glob** 找 `excel/csv/{Table}.csv`；用 **Read** 或 **Grep** 看表头与数据行；禁止未打开文件就写「已验证」。

列类型、ParseType、建表流程：需要时加载 **`.cursor/skills/config-csv-table/SKILL.md`**（仍属「表能否按规范建」，不是架构评审）。

## 文档层级与输入质量（首轮摘要，可选）

若存在下列情况，在**首轮**与评审摘要一并写出（**不占**「单条澄清」额度；属事实描述，一般无需 A/B）：

| 问题类型 | 说明 |
|----------|------|
| 策划原文过薄 | 如策划案仅为占位、缺主流程/边界/规则，标出并建议补材料 |
| 元数据缺口 | 如 SR 已 Approved 但评审人/审核字段为空等，提醒补全以免审计断档 |

## 澄清交互协议（强制）

**目标**：避免**一次性**抛出大量待澄清项与大量选项，降低用户认知负担；用**多轮对话**，每次**一条**、带**结构化选项**。

### 规则

1. **内部**：评审时仍须把全部待澄清项**编号入队**（C-01…C-N），保证不遗漏。
2. **对用户的每条回复**（澄清未完成时）：
   - **最多展示 1 条**当前待澄清项；
   - 必须给出 **选项**，默认 **A / B**；若确有第三种及以上合理语义，再增加 **C、D**，**不宜超过 4 个选项**；
   - 每条选项用**一两句**说清差异与对协议/客户端/验收的影响（见示例）；
   - 文末写：**「请回复选项字母（如 A）；共 N 条待澄清，当前第 k 条。」**
3. **禁止**：在同一条消息里列出 **多条** 待澄清 + 多组 A/B/C/D；禁止「十几条问题请逐条回答」而无选项引导。
4. **用户选定后**：用一行记录「**C-k 结论：选 X，含义：…**」，再发**下一条**（仅一条 + 选项）。若用户回复「跳过/暂不决定」，记入「未决」并仍阻塞可定稿。
5. **全部澄清结束后**：再输出**最终结论**（可否定稿、必改项、修订文档建议）；此时可把**完整澄清结论表**作为附录列出一次，便于归档。

### 单条澄清消息模板（每次一条）

```markdown
### 待澄清 k / N：〈短标题〉

**冲突 / 缺口**：〈一两句：哪两份材料或哪两段规则不一致，或缺什么定义〉

**请选一**（回复字母即可）：
- **A**：〈方案 A 及影响〉
- **B**：〈方案 B 及影响〉
- **C**：（可选）〈方案 C 及影响〉
- **D**：（可选）〈方案 D 及影响〉

---
共 **N** 条待澄清，当前第 **k** 条。选完后将给出下一条。
```

**示例（风格参考）**：过期后仅附件不可领 vs 整封不可读 → **A** 全文不可读仅列表展示；**B** 正文可读、附件不可领（常见）。

## 首轮输出模板（有待澄清时）

首轮在澄清未完成前**不要**给「可定稿」。结构如下：

```markdown
## 评审对象
- 文件：...
- 对照需求（SR 或说明）：...

## 评审摘要（非交互部分）
### 与需求对照
- 已覆盖：...
- 遗漏或冲突（事实）：...

### 策划逻辑与可行性（事实）
- 自洽之处：...
- 已识别的矛盾 / 断点（仅列**条数与编号**，不写展开选项）：共 N 条（C-01…C-N）

### 文档层级与输入质量（若有）
- ...

### 数据来源与 CSV 验证
- 已对齐：...
- 缺失或不一致：...

### 必改项（非选项类，若有）
1. ...

---

## 结论（澄清进行中）
- [ ] 待澄清 **N** 条；**未完成前不可定稿**。

〈接「单条澄清消息模板」，仅 C-01〉
```

澄清全部完成后，再用**最终结论模板**（可含完整澄清结论表附录）。

## 最终结论模板（澄清已全部完成或 N=0 时）

```markdown
## 结论
- [ ] **可定稿**
- [ ] **需修订文档**（必改项：...）
- [ ] **需补充澄清**（若仍有未决项）

## 澄清结论汇总（附录）
| 编号 | 结论（所选选项或用户说明） |
|------|---------------------------|
| C-01 | 选 B：... |

## 与需求对照 / CSV / 必改项
- （摘要或指向前轮）
```

**结论规则**：

- 文档引用某 CSV/列但**文件或列不存在**且未说明待建 → **不得**勾选「可定稿」。
- 存在**未决澄清**（队列未走完或用户跳过未决）→ **不得**勾选「可定稿」。
- 存在**策划逻辑断点或规则自相矛盾**且未通过澄清闭环 → **不得**勾选「可定稿」。

## 常见错误

| 错误 | 正确做法 |
|------|----------|
| 评存储、架构、代码 | 本技能只评**需求与数据来源**；其他用专门评审 |
| 把策划说明当表已落地 | 以 **`excel/csv`** 存在为准 |
| 未读 CSV 就写「列存在」 | 必须对照表头/数据行 |
| 结论含糊 | 必须勾选结论项并写必改项 |
| 只核对 SR 条目不走路径 | 策划逻辑须**按流程走查**，防「写了但不闭环」 |
| 发现缺口不写进待澄清表 | 须在**内部队列**编号，并在交互中逐条消化 |
| **一次性列出全部澄清 + 全部选项** | 违反「澄清交互协议」；**每次只发一条** |
| 选项超过四个或冗长段落 | 控制 **A–D**、每条一两句，保持可选 |
