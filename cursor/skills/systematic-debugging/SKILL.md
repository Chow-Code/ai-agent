# 系统化调试

## 概述

随机改代码浪费时间并制造新 bug；表面修补掩盖根因。

**核心原则：** **先找根因再谈修复。** 只治症状 = 失败。  
**钻流程空子 = 违反流程精神。**

## 铁律

```
未完成第 1 阶段根因调查，不得提出修复
```

## 何时使用

任何技术问题：测试失败、生产 bug、异常行为、性能、构建、集成。

**尤其：** 时间紧、「先 quick fix」很诱人、已试过多种修复、上一修无效、尚未理解问题时。

**不要因**「看起来简单」或「赶进度」而跳过。

## 四阶段（须顺序完成）

### 阶段 1：根因调查

**在尝试任何修复之前：**

1. **细读错误信息** — 堆栈、行号、错误码  
2. **稳定复现** — 步骤、是否必现；不能复现则先收集数据，不要猜  
3. **查近期变更** — diff、依赖、配置、环境  
4. **多组件系统：** 在边界加诊断日志，先看清**断在哪一层**再深入  
5. **数据流追踪：** 深层栈错误见本目录 `root-cause-tracing.md` — 从坏值往上追到源头，在**源头**修  

### 阶段 2：模式分析

- 找类似**正常**代码对照  
- 若照抄模式，把参考实现**读全**  
- 列出与正常例子的**每一处**差异  
- 理清依赖与假设  

### 阶段 3：假设与验证

- **单一假设**：「我认为根因是 X，因为 Y」  
- **最小改动**验证，一次只变一个变量  
- 不成立则**新假设**，不要堆叠多个未验证修复  
- 不懂就说不懂，去问或去查  

### 阶段 4：实现

- **先**有最小失败用例/测试（见 `test-driven-development`）  
- **一次**一个修复，不搞「顺手重构大礼包」  
- 验证：原问题消失、无回归  
- **若修复无效：** 计数已尝试次数；<3 回阶段 1 用新信息重分析；**≥3 次仍失败：怀疑架构/基本假设，与人类协作者讨论后再动**  

## 危险信号 — 停，回到阶段 1

- 「先 quick fix，以后再查」  
- 「改个 X 试试看」  
- 「一次改多处再跑测试」  
- 「不写测试，我手测一下就行」  
- 「多半是 X」但未追踪数据流  
- 未调查就列「主要问题」和一堆修复  
- **已失败 2 次还想「再试一次」**  
- **每次修都在新地方冒新问题**  

**失败 ≥3 次：** 考虑架构问题，勿再盲试。

## 合理化借口

| 借口 | 事实 |
|------|------|
| 「问题简单不用流程」 | 简单也有根因；流程对简单 bug 也很快 |
| 「紧急没时间」 | 系统化比乱猜更快 |
| 「先试试这个再查」 | 第一次修复就定调；一开始就要对 |
| 「确认好了再写测试」 | 无测试的修复站不住 |
| 「一次改多处省时间」 | 无法隔离何者生效，易引入新 bug |
| 「参考太长我改编一下」 | 半懂必出 bug，要读全 |
| 「我看出来了，直接修」 | 见症状≠懂根因 |

## 快速对照

| 阶段 | 关键 | 完成标准 |
|------|------|----------|
| 1 根因 | 读错、复现、变更、证据、追踪 | 理解**什么**与**为什么** |
| 2 模式 | 找正常例、对比 | 找出差异 |
| 3 假设 | 单假设、最小验证 | 证实或换新假设 |
| 4 实现 | 失败测试、单修复、验证 | 问题解决且测试绿 |

## 「没有根因」时

若调查后确属环境/偶发/外部：记录已排除项，再加超时/重试/监控等；但多数「找不到根因」是**调查不完整**。

## 真实影响

系统化通常更快、一次修利率高、少引入新 bug；乱试往往数小时打转。
