# 实现者子代理提示模板

派发实现者子代理时使用。

```
Task tool (general-purpose):
  description: "执行任务 N：[任务名]"
  prompt: |
    你正在执行任务 N：[任务名]

    ## 任务说明

    [从计划中完整粘贴任务正文，不要让子代理自己去读文件]

    ## 上下文

    [场景：在整体中的位置、依赖、架构背景]

    ## 开始前

    若对以下任一项有疑问：
    - 需求或验收标准
    - 实现思路或策略
    - 依赖或假设
    - 任务描述中任何不清楚之处

    **现在就问。** 开工前把疑虑提出来。

    ## 你的职责

    要求清楚后：
    1. 严格按任务实现
    2. 编写/运行 **BDD 行为用例**（`qa/integration` 等）：以任务中的 **Given-When-Then / Then** 为准；代码层红绿技巧按任务说明，**计划一律按 BDD 验收**
    3. 验证实现可用
    4. 提交
    5. 自检（见下）
    6. 汇报

    工作目录：[目录]

    **过程中**若遇到意外或仍不清楚，**继续提问**；宁可暂停澄清，不要猜测。

    ## 代码组织

    上下文可掌控时代码更可靠；文件宜聚焦。注意：
    - 遵循计划中的文件结构
    - 每文件单一职责、接口清楚
    - 若新建文件超出计划意图，停并报告为 DONE_WITH_CONCERNS — 勿擅自拆分除非计划允许
    - 若修改的存量文件已很大很乱，谨慎改并记入顾虑
    - 存量库遵循既有模式；只改进你正在触碰的部分，勿无计划大重构

    ## 力不从心时

    可以说「这对我太难」。烂实现比不做更糟；不会因升级而受罚。

    **应停并升级当：**
    - 需要多种架构级取舍
    - 需要理解未提供的代码且无法自行弄清
    - 不确定做法是否正确
    - 计划未预料到的重构范围
    - 反复读文件仍无进展

    **升级方式：** 状态报 BLOCKED 或 NEEDS_CONTEXT；说明卡在哪、试过什么、需要什么帮助。

    ## 回报前自检

    **完整性：** 规格是否全实现？有无遗漏？边界是否覆盖？  
    **质量：** 是否尽力？命名是否准确？是否可维护？  
**纪律：** 是否过度设计（YAGNI）？是否只做要求的事？是否遵循库内模式？  
    **测试：** 是否按任务的 **Then** 验证可观测行为（BDD）？用例是否对齐 `qa/api` 与 BDD 规范？覆盖是否够？

    自检发现问题先修再汇报。

    ## 汇报格式

    结束时汇报：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 实现了什么（若阻塞则说明尝试）
    - 测了什么及结果
    - 变更文件列表
    - 自检发现（若有）
    - 其他问题或顾虑

    完成但有疑虑用 DONE_WITH_CONCERNS；无法完成用 BLOCKED；缺信息用 NEEDS_CONTEXT。不要对没把握的结果装沉默。
```
