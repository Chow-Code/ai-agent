# 编写实施计划

## 铁律：计划一律 BDD

本技能产出的**每一份**实施计划必须采用 **BDD（行为驱动）**：任务与步骤用 **Given-When-Then**（或与 `qa/integration` 一致的 BDD 用例结构）写清**可观测行为与验收（Then）**，再落到可执行用例（如 `qa/integration`）。禁止把计划写成纯 TDD 套话（例如只写「写失败测试」而无 Then）。代码层是否再用红/绿技巧，由任务内说明，但**计划表述以行为为准**。

## 概述

写**可交给陌生人执行**的完整计划：假设执行者对代码库零上下文、工具不熟。写清每个任务要动哪些文件、如何测、参考哪些文档。任务拆成一口大小。DRY、YAGNI、**一律 BDD**：用 **Given-When-Then** 写清可观测行为，再落到 `qa/integration` 等可执行用例；可按「先失败场景 → 实现 → 再跑通」或「先正例再补反例」小步推进，**每步对齐业务 Then**（详见「任务粒度」）。频繁提交。

假设对方是合格开发者，但几乎不了解你们的工具栈与领域。假设对方测试设计能力一般。

**开始时宣告：**「我使用 writing-plans 技能编写 **BDD 实施计划**。」

**上下文：** 宜在独立 worktree 中运行（由 brainstorming 等工作流创建）。

**保存路径：** `docs/agents-team/plans/YYYY-MM-DD-<功能名>.md`  
（用户另有约定则以用户为准）

## 范围检查

若规格覆盖多个独立子系统，应在 brainstorming 阶段已拆成子项目规格。若未拆，建议拆成**多份计划**——每子系统一份，每份都能单独交付可测软件。

## 文件结构

定义任务前先列出将创建/修改的文件及职责，在此锁定拆分边界。

- 单元边界清晰、接口明确；每文件单一职责。  
- 优先小而专的文件。  
- 一起变的代码放一起；按职责分，不是按技术层硬拆。  
- 存量代码遵循既有模式；若正在改的文件已臃肿，可在计划中包含拆分。

## 任务粒度（BDD：行为先行，本技能强制）

**每一步应是一件小事（约 2–5 分钟），且应对齐「行为」而非仅对齐「测试形式」：**

- **写明行为场景** — 一步（Given-When-Then，或与本项目 `qa/integration` BDD 风格一致的描述；说明要验证的**可观测结果**）  
- **把场景落成可执行用例** — 一步（接口/集成测试代码，或本任务约定的可运行检查方式）  
- **运行用例，确认当前与期望行为的差距** — 一步（可先「红」：行为未满足；也可直接写**正例**验证已存在行为，视任务而定）  
- **写最少实现 / 最小改动，使 Then 成立** — 一步（行为满足即可，避免超写）  
- **再次运行用例** — 一步（正例通过；需要时加**反例/失败路径**场景，同样用行为语言描述）  
- **提交** — 一步  

说明：BDD 强调**行为与验收标准**；「先写会失败的场景」与「先写期望通过的正例」都可以在计划里出现，只要每一步能说清 **When 做什么 → Then 应观察到什么**。

## 计划文件头（必须）

```markdown
# [功能名] 实施计划（BDD）

> **给代理执行者：** 必选子技能：`agents-team:subagent-driven-development`（推荐）或 `agents-team:executing-plans`，按任务逐步执行。步骤用 `- [ ]` 勾选跟踪。**本计划为 BDD：每个任务含 Given-When-Then 或等价可断言 Then。**

**目标：** [一句话]

**架构：** [两三句]

**技术栈：** [关键库/技术]

**测试与验收：** BDD（`qa/integration` / `qa/api/{domain}.yaml`），Given-When-Then，Then 须可客观断言

---
```

## 任务结构示例

每个任务写清：文件（创建/修改/测试路径）、分步勾选、每步命令与**期望输出**、完整代码片段（不要只写「加校验」）。  
涉及测试时，优先写清 **Given / When / Then**（或与 `qa/api/{domain}.yaml`、BDD 用例标题一致），再贴示例代码。

## 记住

- 路径写全、命令写全、期望输出写全；**行为（Then）可客观断言**  
- 用 `@` 引用相关技能（如 `bdd.mdc`、`bdd-qa`）  
- DRY、YAGNI、**BDD 行为小步**、频繁提交  

## 计划评审循环

写完完整计划后：

1. 派发 **plan-document-reviewer** 子代理（见 `plan-document-reviewer-prompt.md`），只给**裁剪后的评审上下文**，不要塞会话历史。  
2. 若有 ❌：修改后**整份计划**再评  
3. 若 ✅：进入执行交接  

超过 3 轮仍卡住：上升人类。

## 执行交接

计划保存后提供二选一：

**「计划已保存到 `docs/agents-team/plans/<文件名>`。执行方式二选一：  
1）子代理驱动（推荐）— 每任务新子代理，任务间评审；  
2）本会话内执行 — 用 executing-plans，分批带检查点。  
选哪种？」**

- 选 1：**必选** `agents-team:subagent-driven-development`  
- 选 2：**必选** `agents-team:executing-plans`
