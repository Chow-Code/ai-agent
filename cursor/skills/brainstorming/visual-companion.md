# 可视化伴侣指南

在浏览器中做头脑风暴：展示线框、示意图、选项对比等。

## 何时用

**按问题决定，不是按会话。** 标准：**用户是「看」更清楚，还是「读」更清楚？**

**用浏览器** —— 内容本身是视觉的：
- UI 线框、布局、导航、组件外观  
- 架构图、数据流、关系图  
- 并排视觉对比（两种布局、配色、方向）  
- 观感、间距、层级等「看起来如何」  
- 空间关系：状态机、流程图、ER 图等  

**用终端** —— 文本或表格即可：
- 需求与范围：「X 指什么」「哪些在范围内」  
- 概念性 A/B/C 文字选项  
- 利弊表、对比表  
- API、建模、架构文字决策  
- 答案主要是文字而非视觉偏好  

**关于 UI 的问题 ≠ 视觉问题。**  
「想要哪种向导流程？」— 概念，用终端。  
「这几个向导布局哪个顺眼？」— 视觉，用浏览器。

## 工作方式（概要）

服务监视目录中的 HTML，把最新页面提供给浏览器；用户点击选项会写入 `.events`，你在下一轮读取。

- 若以 `<!DOCTYPE` 或 `<html` 开头，按完整文档提供；否则服务器会套通用模板。  
- **默认写内容片段即可**，除非需要完全控制整页。

## 启动（示例）

```bash
scripts/start-server.sh --project-dir /path/to/project
```

保存返回的 `screen_dir`，让用户打开 URL。连接信息也可能在 `$SCREEN_DIR/.server-info`。  
使用 `--project-dir` 可把 mockup 存到项目下 `.agents-team/brainstorm/`，便于持久化；勿忘将 `.agents-team/` 加入 `.gitignore`。

**Windows：** 可能需后台运行 Bash，避免前台阻塞。

## 与终端分工

每个问题单独判断：能一句话说清用终端；需要并排看图用浏览器。用户同意伴侣 ≠ 每个问题都开浏览器。

## 更多细节

平台差异、事件文件格式、调试等以项目内 `scripts` 与仓库说明为准；若与本指南冲突以实际脚本为准。
