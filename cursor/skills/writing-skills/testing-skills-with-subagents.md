# 用子代理测试技能

**何时读：** 新建或修改技能、部署前，用于在压力下验证技能是否生效、能否抵抗合理化。

## 概述

**测技能 = 对流程文档做 BDD（行为驱动）。**  
无 SKILL 跑场景（基线失败）→ 写 SKILL（绿）→ 堵洞（重构）。

**核心：** 若没见过「无技能时代理怎么错」，就不知道技能是否防对了。

**前置：** 理解 `test-driven-development`（代码侧）；技能侧用 **Given-When-Then** 或等价压力场景（见 `writing-skills/SKILL.md`）。  
**完整案例（英文）：** `examples/CLAUDE_MD_TESTING.md`（历史文件名，内容仍以场景为主）。

## 何时测技能

**适合：** 强调纪律、有合规成本、易被「就一次」绕过、与短期目标冲突的技能。  

**不必：** 纯参考（API 文档）、无可违反规则、代理无动机绕过的技能。

## 与 BDD / 红绿重构的对应

| 阶段 | 技能测试 |
|------|----------|
| 基线（红） | 无技能跑场景，记录失败与借口 |
| 绿 | 写最小 SKILL，针对基线失败 |
| 重构 | 新借口 → 补条款 → 再测 |

## 红阶段：基线

- 设计**压力场景**（建议多种压力叠加：时间、沉没成本、疲劳、权威等）  
- **不要**加载被测技能，给真实任务式提示  
- **逐字记录**选择与借口  

示例结构：已实现大量代码、忘写测试、快下班、明天评审 —— 选项含「先提交明天补测」等，看代理是否违规。

## 绿阶段：写最小技能

只针对基线里**真实出现**的失败写条款，勿为假想问题堆砌。

再跑**同一**场景**带**技能，应合规；仍失败则技能不清或不全，改后再测。

## 压力场景要点

- **差场景：**「要实现功能，技能怎么说？」— 太学术，只会背条文。  
- **好场景：** 具体后果、真实路径、必须选 A/B/C、要求**行动**而非作文。  
- **更好：** 多种压力组合（见 `writing-skills/SKILL.md` 中的压力类型表思路）。  

开场可写：`IMPORTANT: 这是真实场景，必须选择并行动。` 让代理当真。

## 重构阶段：堵洞

代理**有技能仍违规** = 像场景回归，要改技能。

- 新借口**原文**记入合理化表  
- 规则里**显式否定**每种借口  
- 补「危险信号」列表  
- 必要时在 `description` 增加「即将违规的症状」  

再测直至高压下仍合规。

## 元测试（绿仍不稳时）

在代理选错后问：「技能要怎么写才能让唯一可接受答案一目了然？」  

区分：是**无视清晰规则**（要加强原则/精神表述）还是**规则本身缺句**（补原文）还是**结构找不到重点**（提前呈现关键点）。

## 部署前检查（摘要）

**红：** 多压力场景、无技能基线、逐字记录。  
**绿：** 最小技能、带技能复测、合规。  
**重构：** 新借口、表与条款、description、再测、元测。

常见错误：未先做红、场景不够压、借口记录含糊、泛泛堵洞、一次通过就停。

## 快速对照

| 阶段 | 成功标准 |
|------|----------|
| 红 | 无技能时违规，借口已记录 |
| 绿 | 有技能时遵守 |
| 重构 | 新借口被堵，仍遵守 |

## 底线

**写技能不写场景验证 = 写代码不写测试。**  
对文档走 **BDD 行为场景 + 基线**，与代码侧 TDD/BDD 纪律一致。
