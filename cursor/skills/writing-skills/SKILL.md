# 编写技能

## 概述

**写技能 = 把 BDD（行为驱动）用在流程文档上。**

在 **Cursor** 中，个人技能通常放在项目 `.cursor/skills/` 或本机约定目录（随团队规范而定）。

你先写**可观测的行为场景**（Given-When-Then，或等价的压力场景），在无 SKILL 基线下看代理是否违规；再写 `SKILL.md`；再验证代理行为符合场景；最后堵新借口（重构）。

**核心原则：** 若没见过「没有技能时代理怎么错」，就不知道技能是否教对了。

**前置：**  
- 代码侧测试先行：理解 `test-driven-development`（红-绿-重构）。  
- 接口/集成测试风格：见仓库 `bdd.mdc`、`qa/integration`。  
- **技能侧：** 用 **BDD 场景**（Given-When-Then + 可断言的 Then）描述「代理应如何表现」，再写文档；不要与代码 TDD 混为一谈，但纪律相同：**先行为失败，再补文档，再验证**。

**官方补充（英文）：** Anthropic 技能撰写参考见 `anthropic-best-practices.md`。

## 什么是技能

技能是**可复用的方法、模式、工具参考**，帮助之后的模型找到有效做法。

**是：** 可迁移的技术、模式、工具说明  
**不是：** 某次解决问题的流水账故事  

## 与 BDD / 基线验证的对应

| BDD / 验证 | 技能 |
|------------|------|
| 行为场景（Given-When-Then） | 子代理压力场景或可断言步骤 |
| 实现 | SKILL.md |
| 基线失败（红） | 无技能时代理违规（记录原话借口） |
| 通过（绿） | 有技能后行为符合场景 |
| 重构 | 堵新漏洞仍保持遵守 |

## 何时创建

**创建：** 技巧不直观、会跨项目复用、模式够通用、他人也能受益  

**不创建：** 一次性方案、别处已有标准文档、项目约定（放 `AGENTS.md` / `.cursor/rules`）、可用正则/校验自动化的规则  

## 技能类型

- **技法：** 有步骤的方法（如 condition-based-waiting）  
- **模式：** 思维方式  
- **参考：** API、语法、工具文档  

## 目录结构

```
技能名/
  SKILL.md        # 主文件（必须）
  其他文件         # 仅当需要：长参考、可复用脚本
```

## SKILL.md 结构（frontmatter）

仅支持 `name`、`description`，合计 ≤1024 字符。

- `name`：字母数字连字符  
- `description`：**第三人称**、只写**何时用**（触发条件、症状），**不要**概括流程步骤（否则模型可能只读 description 跳过正文）  
- 推荐以「在……时使用」开头，写具体触发与情境  

正文建议含：概述、何时用、核心模式、快速对照、常见错误、可选真实收益。

## 发现性（description / 关键词）

- **description** 要回答：「我现在该不该读这个技能？」  
- **关键词：** 错误信息、症状、工具名，便于在 Cursor 中检索  
- **命名：** 动名词开头、表意清晰（如 `condition-based-waiting`）  
- **省 token：** 高频加载的技能要短；细节移到 `--help` 或独立参考文件  

## 流程图

仅用于：非显而易见的分支、易早停的循环、「A vs B」决策。  
不要用流程图代替表格、大段代码、线性步骤。

## 铁律（与 BDD 场景先行一致）

```
没有先失败的行为场景 / 基线验证，就没有新技能（也没有未经测试的编辑）
```

先写技能再测？删了重来。  
「只是小改动」也不例外。

## 技能类型与测法概要

- **纪律类（测试先行、完成前验证）：** 场景题 + 高压组合 + 可断言 Then，成功标准：高压下仍遵守  
- **技法类：** 应用场景 + 变体 + 信息缺失，成功标准：能正确应用  
- **模式类：** 识别 + 应用 + 反例，成功标准：知道何时不用  
- **参考类：** 检索 + 应用 + 缺口，成功标准：能找到并正确使用  

详见 `testing-skills-with-subagents.md`。

## 堵合理化

技能要写清：**禁止的借口**与**对应事实**表格、**危险信号**列表、必要时加「违反字面即违反精神」。

## 部署前检查（摘要）

**基线：** 无技能跑场景，记录失败与借口原话。  
**绿：** 写最小技能，再跑同场景应遵守。  
**重构：** 新借口 → 补条款 → 再测。

质量：小流程图仅必要时、快速对照表、常见错误、非故事叙述。

部署前**每个**技能单独测，不要批量未测就发。

## 底线

**写技能 = 对流程文档做 BDD（行为先行 + 验证）。**  
你对代码用 TDD/BDD，对技能用**场景先行 + 基线验证**，同一纪律不同载体。
