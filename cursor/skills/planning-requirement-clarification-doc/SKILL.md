# 策划需求澄清文档生成

## 本技能解决什么

将策划原文（`.txt` / Excel 导出文本，`#` 大节、层级编号、制表符分列）**结构化**为 `{领域}-需求澄清文档.md`：**识别功能点** → **大模块 → 功能点目录树** → 每点 **需求 / 澄清需求 / 边界**。

- **需求**：从原文摘录，清理制表符，**不改语义**。
- **澄清需求**、**边界**：对原文中**不明确、有歧义、缺上下限或缺与外部系统分界**之处，**不得**仅靠模型臆测写死结论；须通过下文 **「澄清交互协议」** 与用户（或产品/策划代表）**确认后**再写入文档。确认后的表述即该功能点的**已定稿澄清**（可标注确认日期或版本一句，由团队约定）。

**长期文档形态**：可与 **`{领域}-开发文档.md`** 双文档并存；开发文档 **DD-1.2** 等以**摘要 + 链接**引用本文件。过渡期其他文档不强制删除。

## 何时使用

- 「生成/更新 策划需求澄清」「从策划原文生成需求澄清文档」「澄清边界」「过一遍需求澄清」
- 已给出策划原文路径或领域名；或需对已生成的 `{领域}-需求澄清文档.md` **补全澄清需求/边界**

## 输入

| 类型 | 说明 |
|------|------|
| **策划原文** | 如 `docs/developdoc/策划原文/{文件名}.txt`；`#` = 大模块；编号与缩进块 |
| **领域目录** | 如 `Y_邮件系统.txt` → `Y-邮件系统`；不确定时在 `docs/design/server/` 搜索已有 `-开发文档.md` / `-需求澄清文档.md` |

## 输出

| 项 | 约定 |
|----|------|
| **路径** | `docs/design/server/{领域目录}/{领域}-需求澄清文档.md` |

### 目录树与每个功能点三项

- **`##` 大模块**、**`### FP-xx 功能点`**：顺序与策划原文一致（见下文「顺序性约束」）。
- **需求**：摘自原文；若涉及**配置表/字段**或**业务判断条件**，写法须与下文「配置与数据来源」一致：**可验证路径** + 行内推荐 **`表名：字段名`**（见下）。
- **澄清需求**：经交互确认后的**待对齐项、歧义消除、口径说明**；未完成确认前可暂标「待澄清」或引用内部队列编号（如 `Q-01`，见交互协议）。
- **边界**：经交互确认后的**范围、上下限、互斥、例外、与其它模块分界**；未完成前可暂标「待澄清」。

**禁止**：在未经用户选择确认前，将虚构的业务规则、数值、枚举含义写入「澄清需求」「边界」并当作已定结论。

### 配置与数据来源（强制）

**目的**：防止文档写了表名/列名，但**仓库里没有对应 CSV** 或**列名不一致**，导致实现与策划「各写各的」。

| 规则 | 说明 |
|------|------|
| **真源路径** | 服务端配置表落地文件位于仓库 **`excel/csv/`** 目录下，例如 `excel/csv/ArenaMatch.csv`；描述「数据来源」时**优先写该相对路径**（与项目根相对），勿写未经验证的别名路径。 |
| **行内记号（推荐）** | 在 **需求 / 澄清需求 / 边界** 中引用**具体字段**时，优先使用 **`表名：字段名`**（**中文全角冒号「：」**）。其中 **表名** 与 `excel/csv/{表名}.csv` 的文件名（**不含 `.csv`**）一致；**字段名** 与该 CSV **表头列名**一致（核验后再写）。**需求里要写判断条件**（如「满足何条件时进入某分支」）时，把条件写在对 **`表名：字段名`** 的取值或比较上，避免口头变量。示例：`当 ArenaMatch：MatchStatus == 1 时…`；首次出现可带路径：`excel/csv/ArenaMatch.csv`，后续同段可用 `ArenaMatch：MatchStatus` 简写（须同一表）。 |
| **写入前核验** | 凡在 **需求 / 澄清需求 / 边界** 中出现**具体表名、字段名、配置项**，执行写入或更新前须在仓库内 **核实**：`excel/csv/{表名}.csv` **是否存在**；若谈列，**表头列名**是否与文档一致（用读取文件或检索工具，**禁止凭记忆断言**）。 |
| **可确认才写死** | 仅当上述核验通过时，方可在正文中将「数据来源」写为确定表述：路径 + **`表名：字段名`** 列表或条件。例：`excel/csv/SendEmail.csv`；字段引用 `SendEmail：ID`、`SendEmail：Type`。核验失败时须写 **明确状态**：如「仓库中暂无 `excel/csv/Xxx.csv`，须先建表或同步策划表」，并可将「是否新增该 CSV」纳入 **澄清队列**（仍遵守每次只问一题）。 |
| **与领域对齐** | 表与业务领域的对应关系以 **`excel/csv`** 实际文件及 **`docs/design/server/{领域}/`** 既有文档为准；跨领域引用表时同样须核验路径存在。 |
| **更深规则** | 列类型、ParseType、建表扩展见 **`.cursor/skills/config-csv-table/SKILL.md`**；与 SR/开发文档/CSV 三方交叉评审见 **requirement-doc-data-source-review**。 |

**禁止**：在未打开或检索 `excel/csv` 的情况下，编造「肯定有某张表」；将策划 oral 名称写成正式列名而不核对表头。

### 功能点拆分粒度（与源文件「功能点」数量对齐）

**目标**：生成文档中的 **FP 个数**、**先后顺序** 与策划原文的 **主题块** 一致；**不**默认把每个子编号拆成独立 FP，以免 FP 数远大于源文件「模块数」、与策划体感不一致。

| 层级 | 说明 |
|------|------|
| **默认（推荐）** | **功能点** = 策划在每个 `#` 大节下，自上而下出现的 **第一级主题块**：通常为**独占行的模块名 + 冒号**（如「邮件添加：」「邮件编辑：」「全服发送：」「邮件复制：」「删除邮件：」…），或 **# 节之间**的独立能力段。**同一主题块内**的 **1.1 / 1.2 / 2.1** 等子编号 **不增加 FP 数量**，全部写入该 FP 的 **需求** 正文（可分层编号列出），**顺序与原文一致**。 |
| **独立标题行必拆** | 策划已写成**两条不同能力**的独立行（如「删除邮件」与「删除全部邮件」、「发送邮件」与「发送全部邮件」），**必须**各为 **一个 FP**，与原文条数一致，**禁止**合并。 |
| **禁止默认过度拆分** | **禁止**在默认模式下将「复制对象」下的 1.1、1.3～1.7 **各拆一个 FP**；应放在 **「邮件复制」** 单一 FP 的 **需求** 中逐条列出。**除非**用户显式要求 **「细粒度」「每个子编号一个 FP」**。 |
| **细粒度模式** | 仅当用户**显式声明**时启用：按子编号拆成多个 FP，此时 FP 数**会多于**默认「主题块」数；须在文首注明「细粒度生成」。 |

**自检**：默认生成后，**FP 数量 ≈ 源文件主题块数量**（同一 `#` 下数「邮件添加/邮件编辑/…」级标题）；**需求** 正文覆盖该块内全部子行，**无漏句**。

### 顺序性约束（强制）

大模块与功能点次序必须与原文**自上而下**阅读顺序一致；**禁止**重排、按主题聚类、为对齐开发文档章节而改策划顺序。在遵守上文「独立标题须拆 FP」的前提下，其余合并须**保守**，子点顺序保持原行序单调。

### 新增与原文外内容（位置固定）

非原文条目**仅**放在文件**最末尾**（如 `## 附录：新增与补充`）。

## 澄清交互协议（强制）

**目标**：对**不明确的需求**、**待定的边界**与用户对齐；**降低认知负担**——**每次只问一个问题**，带**结构化选项**；**禁止**在一条回复里罗列多道澄清题让用户一次性消化。

### 规则

1. **内部（可全量）**：通读原文与已生成的需求澄清文档，将待澄清点合并为**有序队列**（建议编号 `Q-01`…`Q-N`，并注明所属 `FP-xx`）。队列用于代理自检与防遗漏，**不**整表甩给用户。
2. **对用户的每条回复**（澄清未完成时）：
   - **最多展示 1 条**当前待澄清项；
   - 必须附带 **A / B / C / D**（或「其它：___」）等**可选答案**；选项应互斥、可落地，避免空泛「再看」；
   - 简要说明该问题对应原文或 `FP-xx` 位置，便于用户理解语境。
3. **用户选定后**：代理将**结论写入** `{领域}-需求澄清文档.md` 中对应功能点的 **澄清需求** 和/或 **边界**（按问题性质），必要时在句首加「已确认：」或团队约定的标记；然后从队列取出**下一题**，仍只发一条。
4. **队列为空**：告知用户本阶段澄清已结束，并提示可定稿或进入 `requirement-doc-data-source-review` / 开发文档同步。

### 生成文档后必须发起澄清（强制）

- **技能 ≠ 只落盘 Markdown**：生成 `{领域}-需求澄清文档.md` 并留下「待澄清」，**只完成了「结构 + 需求摘录」**；**澄清交互尚未发生**，除非用户已在对话里逐条回答。
- **同一会话内必须开口问**：若本次写入（或更新）的文档中存在 **待澄清**（或等价占位），且用户**未**事先声明「只出骨架、不要提问」「只生成文件、不澄清」，则代理在**写完文件的同一轮回复末尾**，**必须**附上 **Q-01**（**仅一条**待澄清项 + **A/B/C/D**）。**禁止**只写文件、不发起任何提问就结束本轮。
- **为何不能指望「大模型都知道」**：澄清需求与边界涉及**产品/策划口径、与 CSV/实现的约定**；模型可推测，但**技能禁止**将未经验证的推测写入「澄清需求」「边界」作为定稿。「待澄清」= **尚未由你（或产品）确认**，不是「模型已心中有数」。

### 全量澄清（用户明确要求时）

若用户表示 **「每个功能点都要澄清」「全量过一遍」「FP 全部过」** 等：将 **FP-01…FP-N** 按原文顺序**全部入队**（每个 FP 至少覆盖 **澄清需求** 与 **边界** 中待定的点；可拆为每 FP 一题或每 FP 两题，由用户在首轮选定粒度）。**纪律不变**：队列再长，对用户仍 **每次只展示一题 + 选项**，逐条落盘后再下一题，直至队列为空。

### 与用户沟通的禁止项

- **禁止**一次性列出「以下 10 条请确认」。
- **禁止**在无选项、无上下文的情况下只问开放式「你觉得呢」代替可执行澄清。
- **禁止**用户尚未选择就把猜测写进「澄清需求」「边界」作为最终结论。

### 与 requirement-doc-data-source-review 的关系

若同时加载 **requirement-doc-data-source-review**，二者**均以「每次一条 + 选项」**为对用户输出澄清时的纪律；本技能侧重 **{领域}-需求澄清文档** 的结构化落盘与策划原文顺序；评审技能侧重 SR/开发文档/**CSV 列与数据行**交叉验证。配置类表述须与 **`excel/csv`** 一致；可共用同一队列编号习惯，避免重复提问同一事项。

## 执行步骤

1. **解析领域与路径**：映射领域目录；必要时在 `docs/design/server/` 搜索已有文档。
2. **读取策划原文**（及已存在的需求澄清文档）：顺序扫描，划分大模块与功能点（**保序**）。
3. **配置数据源核验（与步骤 4 可交错）**：对策划原文中出现的配置表名/字段，在 **`excel/csv/`** 下 **Glob/Read** 核对文件与表头；结论用于步骤 4 的「可确认数据源」表述。
4. **生成或更新文档骨架（保序）**：输出 `##` / `###`；**需求**列写摘录，涉及配置与判断条件时**仅写已核验的** `excel/csv/...`，字段与条件用 **`表名：字段名`** 表达；对尚不清楚的 **澄清需求**、**边界** 标 `待澄清` 或登记入内部队列 `Q-xx`（**不**用编造正文填满）。
5. **执行澄清循环**：按队列顺序，对用户**每次只发一题 + 选项**；用户回答后**立即**更新 Markdown 对应 `FP` 的 **澄清需求** / **边界**，再发下一题。
6. **文首元信息**：来源路径、日期、状态；澄清推进后可更新状态（如 Reviewing / Approved 由团队定）。
7. **噪声**：`DISPIMG` 等——需求中可标注「见策划附图」；边界不臆造。
8. **新增置尾**：原文未覆盖的条目仅追加**文末**附录。
9. **发起澄清**：若文档含「待澄清」且用户未豁免提问，**同一轮回复**须包含 **Q-01**（一条 + 选项）。

## 验收（代理自检）

- 文档结构顺序与原文一致。
- 每个功能点三项齐全；**澄清需求**、**边界**中凡非「待澄清」的句子，均应能对应到**用户某次明确选择**（或用户书面给出的结论），而非模型独白。
- 对用户的澄清轮次中，**从未**出现单条回复含多条待确认题。
- 附录仅在文末。
- **配置可验证**：文中出现的每个 `excel/csv/*.csv` 路径与 **`表名：字段名`** 中的字段名，均已在本轮会话中通过仓库文件核对，或已明确标注「暂无该文件/待建表」等状态；**禁止**未核验却写「以某表为准」。
- **澄清已发起**：若文中仍有「待澄清」且用户未要求「只出文件不提问」，则本轮对话**已**输出 **Q-01**（或用户已声明暂缓澄清）；**禁止**仅写文件而不给任何选项。

## 不做什么

- 不从 `proto`/`qa/api` 反推需求；不替代开发文档技术设计章节。
- 不自动删除其它设计文档；不自动跑 SVN。
