# 接收代码评审

## 概述

代码评审是技术判断，不是情绪表演。

**核心原则：** 先核实再改。拿不准就问。技术正确优先于社交舒适。

## 响应模式

```
收到代码评审意见时：

1. 读：完整读完，不急着反应
2. 懂：用自己的话复述要求（或提问）
3. 核：对照代码库事实
4. 评：对**本仓库**是否技术成立？
5. 回：技术确认或有理有据的反对
6. 改：一次一项，每项后测试
```

## 禁止的回复

**绝不：**
- 「您说得完全对！」（显式违反协作规范类说明时）
- 「好点子！」「反馈很棒！」（表演式）
- 「我这就改」（核实之前）

**应改为：**
- 复述技术要求  
- 问澄清问题  
- 若不对，用技术理由反对  
- 少废话多动手  

## 反馈不清时

```
若任一项不清楚：
  停 — 先不要改任何一项
  问 — 对不清楚项要求澄清

原因：各项可能关联。一知半解 = 改错。
```

## 按来源处理

### 来自人类协作者
- 一般可信 — 理解后再改  
- 范围不清仍要问  
- 不要表演式附和  
- 直接行动或技术确认即可  

### 来自外部评审者

**落实前先检查：**
1. 对**本仓库**是否技术上正确？  
2. 是否破坏现有行为？  
3. 当前实现是否有历史原因？  
4. 是否覆盖所有平台/版本？  
5. 评审者是否了解完整上下文？  

若建议明显不对：用技术理由反对。  
若难以核实：说明「没有 X 我无法核实，应调查/询问/还是继续？」  
若与协作者先前架构决策冲突：先与协作者对齐。  

### 「专业」功能的 YAGNI 检查

若评审建议「正规实现」：先在代码库里搜实际用法。  
未使用：「该端点无人调用，是否删除（YAGNI）？」  
有使用：再谈正规实现。

## 落实顺序

多条款反馈时：
1. **先**澄清所有不清楚项  
2. 再按：阻塞项（崩溃、安全）→ 简单修复 → 复杂修复  
3. 每项单独测试  
4. 回归检查  

## 何时反对

当建议：破坏现有功能 / 缺少上下文 / 违反 YAGNI / 对本栈不正确 / 有兼容或遗留原因 / 与协作者架构决策冲突。  

**反对方式：** 技术论证、具体问题、引用测试/代码；架构问题拉协作者。

## 承认正确反馈

正确时：
- ✅「已修。[简述变更]」  
- ✅「[具体问题] 在 [位置] 已修。」  
- ✅ 或直接改代码展示  

避免空洞感谢；代码即回应。

## 若先前反对错了

- ✅「核实后 [X] 确实 [Y]，正在改。」  
- ❌ 长篇道歉或辩解  

## 常见错误

| 错误 | 纠正 |
|------|------|
| 表演式附和 | 复述要求或直接改 |
| 盲目照做 | 先对照代码库 |
| 批量改不测 | 一项一测 |
| 预设评审永远对 | 检查是否破坏行为 |
| 该反对不反对 | 技术正确 > 舒适 |
| 部分落实 | 先澄清全部条目 |
| 无法核实仍继续 | 说明限制并问方向 |

## GitHub 线程

回复行内评审评论时，用评论线程 API 回复，勿仅发顶层 PR 评论（若工作流如此要求）。

## 底线

**外部反馈 = 待评估的建议，不是必须执行的命令。**  
核实、质疑，再落实。不要表演式附和，始终技术严谨。
