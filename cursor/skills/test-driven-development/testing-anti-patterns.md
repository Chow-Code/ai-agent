# 测试反模式

**何时读：** 编写或修改测试、加 mock、或想在生产类上加「仅测试用」方法时。

## 概述

测试应验证**真实行为**，不是 mock 的行为。Mock 用于隔离，不是被测对象。

**核心原则：** 测代码做什么，不测 mock 做什么。  
**严格 TDD 能避免下列反模式。**

## 铁律

```
1. 绝不测 mock 的行为
2. 绝不给生产类加仅测试用的方法
3. 未理解依赖前绝不乱 mock
```

## 反模式 1：测 mock 行为

**错误：** 断言页面上存在 `sidebar-mock` 等 —— 只证明 mock 在，不证明组件对。

**修复：** 测真实组件行为；若必须 mock 侧栏，则断言页面级行为，而非 mock 元素存在。

**门禁：** 断言任何 mock 前问：「我在测真实行为还是 mock 存在？」若是后者，删断言或取消 mock。

## 反模式 2：生产类里的仅测试方法

**错误：** 为测试加 `destroy()` 等，生产从不调，却像正式 API。

**修复：** 清理逻辑放在 `test-utils` 等测试工具函数里，生产类保持干净。

**门禁：** 加方法前问：「是否仅测试用？」是则不放生产类；再问：「生命周期是否归该类管？」否则放错类。

## 反模式 3：不理解就 mock

**错误：** mock 掉测试仍依赖的副作用（如写配置），导致测逻辑断裂。

**修复：** 先弄清真实方法副作用；测依赖哪些副作用；在**更底层** mock 慢/外部部分，保留测试所需行为；不确定时先用真实实现跑通再最小化 mock。

**红旗：** 「先 mock 保险」「可能慢就 mock」、未理清依赖链就 mock。

## 反模式 4：不完整 mock

**错误：** 只造「当前断言用到的字段」，下游却读别的字段 —— 集成时静默失败。

**铁律：** mock 数据结构应贴近真实 API/schema，不仅是眼前两行断言用到的字段。

**门禁：** 查真实响应有哪些字段；下游可能消费的都带上；不确定则按文档列全。

## 反模式 5：测试是事后补充

**错误：** 实现标完成却没有测试。

**修复：** TDD：红 → 绿 → 重构 → 再声称完成。

## Mock 过于复杂时

征兆：mock 比测试本体还长、为通过而全 mock、mock 缺真实组件的方法、改 mock 就炸。

**问：**「这里是否真的需要 mock？」真实组件的集成测试往往更简单。

## TDD 如何防这些

1. 先写测试 → 想清楚到底测什么  
2. 先见失败 → 确认在测真实行为  
3. 最少实现 → 不易混入仅测试 API  
4. 先见真实依赖再决定 mock  

**若在测 mock 行为，已违反 TDD。**

## 快速对照

| 反模式 | 对策 |
|--------|------|
| 断言 mock 元素 | 测真实组件或取消 mock |
| 生产类仅测试方法 | 移到测试工具 |
| 不理解就 mock | 先理解依赖，最小 mock |
| 不完整 mock | 对齐真实结构/schema |
| 测试后补 | 测试先行 |
| mock 过复杂 | 考虑集成测试 |

## 危险信号

- 断言 `*-mock` 等测试 ID  
- 方法只被测试调用  
- mock 占测试大半篇幅  
- 去掉 mock 测试行为就变了  
- 说不清为何需要 mock  
- 「先 mock 保险」

## 底线

**Mock 是隔离工具，不是被测目标。**  
若 TDD 暴露出你在测 mock，路线已偏。  
修正：测真实行为，或重新审视是否必须 mock。
