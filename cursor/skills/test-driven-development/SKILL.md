# 测试驱动开发（TDD）

## 概述

先写测试 → 看失败 → 写最少代码通过。

**核心原则：** 若没见过测试失败，就不知道它是否在测正确的东西。  
**钻规则空子 = 违反规则精神。**

## 何时使用

**默认始终：** 新功能、修 bug、重构、行为变更。

**例外（需人类同意）：** 一次性原型、生成代码、纯配置。

想「就这一次跳过 TDD」？停，那是合理化。

## 铁律

```
没有先失败的测试，就没有生产代码
```

先写了实现？删掉，重来。不要保留当「参考」、不要边写测边「改编」——**删 = 删**。

## 红-绿-重构

1. **红：** 写一个**失败**的、最小的、表达期望行为的测试。  
2. **核实红：** 运行测试，必须**失败得对**（缺功能而非笔误）。  
3. **绿：** 写**最少**代码使测试通过。  
4. **核实绿：** 全绿、无意外警告。  
5. **重构：** 仅在绿后整理，不加行为。  
6. 重复下一行为。

**好测试：** 单行为、命名清楚、尽量测真实代码（避免无谓 mock）。  
**坏测试：** 名字含糊、只测 mock。

## 为何顺序重要

「写完再补测试」—— 测试一上来就绿，**不能证明测到了该测的东西**；可能测错对象、测实现不测行为、漏边界。  
先失败证明测试**真的会抓问题**。

## 常见借口

| 借口 | 事实 |
|------|------|
| 「太简单不用测」 | 简单也会坏；写测只要几十秒 |
| 「写完再测」 | 先写代码再测，通过无证明力 |
| 「手测过了」 | 手测不可重复、无记录 |
| 「删了 X 小时太浪费」 | 沉没成本；留着不可信代码才是债 |
| 「TDD 教条，我务实」 | TDD 才是务实：早失败、防回归、可重构 |
| 「保留参考边写测」 | 那是测后写，删干净重来 |

## 危险信号 — 停，重来

- 先写实现再写测试  
- 测试第一次跑就绿（未先见红）  
- 「先手动验证再写测试」  
- 「spirit not ritual」式辩解  

**= 删实现，从 TDD 重来**（除非人类明确豁免）。

## 完成前检查

- [ ] 每个新公开行为有测试  
- [ ] 每个测试在实现前**见过红**  
- [ ] 失败原因符合预期（缺功能）  
- [ ] 实现是最少的  
- [ ] 全绿、输出干净  
- [ ] 尽量真实依赖，mock 仅必要时  

## 卡住时

| 问题 | 方向 |
|------|------|
| 不知怎么测 | 先写理想 API/断言；问人类 |
| 测试太难写 | 往往设计太难，简化接口 |
| 必须全 mock | 耦合过重，考虑注入 |
|  setup 巨大 | 抽 helper；仍复杂则简化设计 |

## 与调试集成

修 bug：**先**写复现失败的测试，再走 TDD；无测试的修复不接受。

## 反模式

见同目录 `testing-anti-patterns.md`：测 mock、给生产类加仅测试用方法、未理解依赖就 mock 等。

## 最终规则

```
生产代码 ⟺ 存在且曾先失败的测试  
否则 ⟺ 不是 TDD  
```

未经人类允许，无例外。
