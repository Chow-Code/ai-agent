# 需求变化同步（Requirement Change Sync）

## 本技能解决什么

**已有功能在迭代或局部调整时，避免「代码/配置改了，但 `docs/design/server/{功能}/` 下的服务器需求文档与配置使用文档没改」**。

- 适用于**已经存在**于 `docs/design/server/` 的功能；新功能首版请走 `requirement.md` + `development.md` 命令。
- 与 `design-doc-sync` 子代理配合：本 skill 做**判定、定位、派发**；写入由子代理完成。
- 配置表新增/重导走 `sync-client-config`、`config-csv-table`；本 skill 只负责把变化**反映到设计文档**。

## 触发场景

满足任一即应启动本 skill：

- 用户说「需求变了 / 策划改了需求 / 这块逻辑改了，同步一下文档」
- 用户说「配置表改了 / 加了字段 / 改了数值规则，同步配置使用文档」
- 已有功能新增 User Story、新增/调整接口、修改业务规则、修改 Redis Key/Storage 字段
- 已有功能的 `excel/csv/{表名}.csv` 表头/枚举/全局参数 `GlobalDefine` 字段发生增删改
- 实现一段变更后，**提交/合并前**自检：设计文档是否落后于代码与配置

**不触发**：

- 全新功能首版（用 `.cursor/commands/requirement.md` + `.cursor/commands/development.md`）
- 纯代码重构、不改对外行为与配置（无需更新设计文档）
- 仅修复实现 bug 且不改协议/存储/规则（除非引入了新错误码或新边界）

## 变化分类（决定写入哪个文档）

| 变化类型 | 关键判断 | 同步目标 |
|----------|---------|----------|
| 业务规则、流程、边界、错误码 | User Story / FR-xxx / 业务流程 / 错误码 / 上下限 | `{功能}-服务器需求.md`（SR-x 对应章节）+ `{功能}-开发文档.md` 相关 DD-x（如 DD-2/DD-3/DD-14） |
| 协议（消息名/字段/Ret） | `MsgCtr*` / `MsgSoc*` 名称、字段、错误码 | `{功能}-服务器需求.md` 协议章节；并核对 `qa/api/{domain}.yaml` 是否需同步 |
| Redis Key / Storage 结构 | `entity/storage/key_constants.go`、`*Storage` 字段 | `{功能}-服务器需求.md` 数据存储章节；如有 ChangeTracker/落库变化需说明 |
| 跨子域依赖、网关接口、事件 | 跨 `domain/` 调用、新增 gateway / 事件订阅 | `{功能}-服务器需求.md` 架构 / 关联系统章节 |
| 配置表字段（CSV 表头）增删改 | `excel/csv/{表名}.csv` 表头变化、`Ct{表名}.go` 字段变化 | `{功能}-配置使用.md` 对应章节；并加 `〇 对账摘要` 一行 |
| `GlobalDefine` 全局参数变化 | `excel/csv/GlobalDefine.csv` 与本功能相关字段 | `{功能}-配置使用.md` GlobalDefine 章节 |
| 枚举（proto）增删 | `EMsg*` 在 `proto/` 中的变化 | 两份文档凡引用处统一更新（不写魔法数字） |
| 配置表新增整张表 | `excel/csv/` 新出现 `{表名}.csv` | `{功能}-配置使用.md` 总览 + 新章节；先用 `config-usage-doc` 技能确认结构 |

## 输入与定位

### 步骤 1：确定功能目录

1. 读取用户输入的功能名（如 `Y-邮件系统`、`C-宠物`、`J-绝学`）。
2. 在 `docs/design/server/` 下定位目录；找不到精确匹配时，按首字母前缀模糊匹配并向用户确认。
3. 列出该目录下文件，至少应存在：
   - `{功能}-服务器需求.md`
   - `{功能}-开发文档.md`（可能没有，本 skill 仅强制要求需求文档；开发文档若存在也要联动更新）
   - `{功能}-配置使用.md`（涉及配置变化时强制要求；不存在时改用 `config-usage-doc` 技能先生成）

若功能目录在 `docs/design/server/` 下**完全不存在**，告知用户：本 skill 用于「同步变化」，新功能请改用 `requirement.md` 命令。

### 步骤 2：归集变更证据

把以下作为变更的真源（按优先级），**禁止**仅凭对话描述凭空写入：

| 优先级 | 证据来源 | 用途 |
|--------|----------|------|
| 1 | 当前会话内用户书面描述 + 已确认的澄清结论 | 业务规则变化、边界、错误码 |
| 2 | 已修改/将修改的代码（Handler / AppService / Domain / Storage） | 协议、存储、错误码、流程 |
| 3 | `proto/` 下 `.proto` 变更 | 协议字段、消息名、枚举 |
| 4 | `excel/csv/{表名}.csv` 与 `excel/code/Ct{表名}.go` | 配置字段变化（**权威**：CSV + Go 代码） |
| 5 | `docs/developdoc/策划原文/{功能名}.md`、`{功能名}/变更记录.md` | 策划口径变更，可作为「输入来源」标注 |
| 6 | `qa/api/{domain}.yaml` | 协议对账 |

### 步骤 3：澄清未明项（同 planning-requirement-clarification-doc 的纪律）

- 凡是用户口头描述模糊（数值、上下限、与其他功能的边界、错误码语义）的，**每次只问一题 + A/B/C/D 选项**；用户选定后再写入。
- **禁止**把未确认的推测当作「已变更结论」写进文档。

## Subagent 派发（核心执行模式）

**主流程不直接大段改文档**，而是把上面归集好的变更证据，**通过 Task 派发给 `design-doc-sync` 子代理**完成写入。多份文档可并行派发。

### 派发模板（每条 Task）

> 子代理可选 `design-doc-sync`（推荐）；若无可用子代理，主流程也可顺序执行同样工作，但仍须遵守本 skill 的增量更新约束。

发送给子代理的 prompt 须包含以下信息（缺一不可）：

1. **功能目录与目标文档绝对路径**（如 `docs/design/server/Y-邮件系统/Y-邮件系统-服务器需求.md`）。
2. **变更类别**（业务规则 / 协议 / 存储 / 配置 / 错误码 / GlobalDefine ……，可多选）。
3. **变更摘要**（一句话）+ **变更明细**（条目化），明细引用真源（代码路径、proto 路径、CSV 路径、字段名）。
4. **必须的写入约束**（参见下文「增量更新约束」）。
5. **禁止事项**：禁止改写未受影响章节；禁止删除历史；禁止编造未在证据中的字段或规则。
6. **产出要求**：返回修改了哪些章节、新增/修改/删除了哪些条目；若有歧义留作 `待澄清` 并回报给主流程。

### 派发并行度

- **服务器需求文档** 与 **配置使用文档** 可**并行**派发为两条 Task（无共享可变状态）。
- **开发文档**（`{功能}-开发文档.md`）若需要联动更新（如 DD-2 新增 User Story、DD-14 新增错误码、DD-4 引用配置变化），作为**第三条** Task；与前两条可并行。
- 三条 Task 完成后，主流程做汇合校验（见「验收清单」）。

## 增量更新约束（强制传递给 subagent）

| 约束 | 说明 |
|------|------|
| **只增量、不重写** | 仅修改受影响段落；其他章节保持原样。**禁止**整篇覆盖。 |
| **保留历史可追溯** | 在被修改的小节末尾或条目尾部追加 `[更新于 YYYY-MM-DD：变更摘要一句话]`；若原内容有意义但被替换，把旧内容移到「附录：变更前内容」并标注日期。 |
| **更新元信息** | `文档元信息` 中 **最后更新** 改为今日；状态从 `Approved` 视情况降为 `Reviewing` 或保持，并在`审核意见`追加一行 `YYYY-MM-DD：xxx`。 |
| **配置使用文档加对账行** | `〇、正式配置对账摘要` 中追加一行：议题 / 历史文档写法 / 正式 CSV 现状 / 服务端注意。 |
| **可验证才写死** | 出现表名、CSV 列名、Go 字段名、proto 字段名时，须能在仓库中找到（`Read`/`Glob`/`Grep` 核验）；找不到则写「待新增」状态而非编造。 |
| **协议字段同步** | 如改了协议，提示主流程同步检查 `qa/api/{domain}.yaml` 是否需更新（不强制由本 skill 完成，但要在产出回报中列出）。 |
| **错误码同步** | 新增错误码须在 `proto/EMsgErrorType.proto` 中存在；不存在时写「待定义」并报回主流程。 |
| **完整消息名** | 文档中消息名一律用全名（如 `MsgCtrReqXxx`），禁止简化。 |

## 与其他技能/命令的边界

| 场景 | 该用什么 |
|------|---------|
| 全新功能首版需求/开发文档 | `.cursor/commands/requirement.md` → `.cursor/commands/development.md` |
| 已有需求做策划口径澄清 | `planning-requirement-clarification-doc`（写到 `{功能}-需求澄清文档.md`） |
| 配置表「使用文档」首次创建 | `config-usage-doc` |
| 新建 CSV / 改 ParseType | `config-csv-table` + `sync-client-config` |
| 已有功能的「需求/配置变化」回写设计文档 | **本 skill** |
| BDD 用例同步 | 本 skill 完成后，提示用户走 `bdd-qa` / `.cursor/commands/test-case.md` |

## 执行步骤总览

1. **判定**：是否属于「已有功能的需求/配置变化」（不是则推荐其它命令/技能）。
2. **定位**：在 `docs/design/server/{功能}/` 下找到目标文档。
3. **分类**：按上文「变化分类」表，区分要写入哪份文档。
4. **归集证据**：按优先级收集真源（代码 / proto / CSV / 策划原文）；含模糊点先一题一题澄清。
5. **派发 Task**：对每份目标文档派发一条 `design-doc-sync` 子代理任务，prompt 须含目标路径、变更类别、变更明细、写入约束、禁止事项、产出要求。
6. **汇合**：收到子代理回报后，按「验收清单」自检；遗漏项再发补充 Task。
7. **联动提示**：列出需用户后续处理的事项（`qa/api/*.yaml`、`proto/EMsg*`、BDD 用例、`gm-command` 等）。

## 验收清单（主流程自检）

- [ ] 仅修改了受影响章节；未受影响章节字数无变化（可对比 diff）
- [ ] 元信息「最后更新」已改为今日，并追加了一行变更摘要
- [ ] 每个修改的小节有 `[更新于 YYYY-MM-DD]` 标记
- [ ] 配置使用文档若改动，已在 `〇、对账摘要` 追加一行
- [ ] 协议变更点已联动列出 `qa/api/{domain}.yaml` 待办（即便本轮未改）
- [ ] 错误码变更已与 `proto/EMsgErrorType.proto` 对账
- [ ] 文中表名/字段名均能在 `excel/csv/` 与 `excel/code/` 找到，否则已显式标注「待新增」
- [ ] 没有把未澄清的推测写入正文当作结论
- [ ] 文档名/路径符合 `docs/design/server/{首字母}-{中文名}/{功能}-服务器需求.md` 与 `{功能}-配置使用.md` 命名

## 禁止事项

- ❌ 一次性整篇重写设计文档（必须增量）
- ❌ 在没有任何代码/配置/策划证据的情况下，按「想当然」写入新规则
- ❌ 把简化消息名（如 `login`）写入文档
- ❌ 写「以某表为准」却没有核对 `excel/csv/{表名}.csv` 是否存在
- ❌ 跳过 subagent 派发，由主流程一口气大段改写多份文档（除非用户明确豁免）
- ❌ 删除文档中已有的历史决策与背景信息
