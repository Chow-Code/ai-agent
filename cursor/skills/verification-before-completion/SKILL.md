# 完成前验证

## 概述

未经验证就声称完成，不是高效，是不诚实。

**核心原则：** 永远先证据，后结论。  
**钻规则空子 = 违反规则精神。**

## 铁律

```
没有本轮新鲜验证证据，不得声称完成/通过
```

若本消息中尚未运行验证命令，就不能声称通过。

## 门禁

```
在声称任何状态或满意之前：

1. 认定：什么命令能证明该声称？
2. 执行：跑**完整**命令（新鲜、完整）
3. 阅读：完整输出、退出码、失败数
4. 核实：输出是否支持该声称？
   - 否：用证据说明实际状态
   - 是：再附带证据作出声称
5. 仅在此之后：才能声称

跳过任一步 = 未验证，而非已验证
```

## 常见失败

| 声称 | 需要 | 不算数 |
|------|------|--------|
| 测试通过 | 测试命令输出 0 失败 | 上次跑的、「应该过」 |
| Lint 干净 | Lint 输出 0 错误 | 只查了一部分 |
| 构建成功 | 构建命令 exit 0 | Lint 过≠编译过 |
| Bug 已修 | 复现症状的原测试/步骤通过 | 只改了代码想当然 |
| 子代理完成 | 看 diff、本地验证 | 轻信「成功」报告 |

## 危险信号 — 停

- 使用「应该」「大概」「看起来」  
- 验证前就满意（「太好了！」「搞定！」）  
- 未验证就要提交/推送/PR  
- 轻信子代理成功报告  
- 部分验证当全部  
- 「就这一次」  
- **任何未跑命令却暗示成功的说法**  

## 合理化借口

| 借口 | 事实 |
|------|------|
| 「现在应该好了」 | 去跑验证 |
| 「我有信心」 | 信心≠证据 |
| 「就这一次」 | 没有例外 |
| 「Lint 过了」 | Lint≠编译 |
| 「代理说成功」 | 独立核实 |
| 「太累了」 | 累不是借口 |
| 「只查了一部分」 | 部分不能证明整体 |
| 「换种说法就不算违规」 | 精神高于字面 |

## 何时适用

**在以下任何情况之前：**
- 任何成功/完成类表述  
- 任何满意情绪  
- 任何暗示工作已正确完成的表述  
- 提交、PR、任务完成、进入下一任务  
- 再派发给代理  

**适用于：** 原话、转述、暗示成功的任何沟通。

## 底线

**验证没有捷径。**  
跑命令 → 读输出 → 再下结论。不可协商。
