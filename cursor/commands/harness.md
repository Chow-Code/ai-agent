# Feature Harness（需求驱动、端到端自动做完）

## 使用方式（用户）

1. **切换到 Cursor Agent 模式**（需能读文件、改代码、跑终端命令）。纯 Ask/Chat 无工具时无法替你「做完」。
2. 通过命令面板运行本命令，或在对话里 **@harness**，后面粘贴你的需求（可含领域、接口范围、验收点）。
3. Agent 应 **连续执行** 下文流程，直到：验证命令有输出证据，或 **唯一阻塞**（缺 PB/客户端对齐信息、缺账号/环境）已写清并列出待你补充的项。

## 用户输入（需求）

```text
$ARGUMENTS
```

若为空，先向用户索要：业务目标、涉及领域（如 `role`/`backpack`）、是否新协议/新消息、验收标准。

---

## Agent 必须遵守的规则（先读后做）

执行本流程前 **必读** 并在步骤中落实：

- `.cursor/rules/harness-always.mdc`（PB 优先、API 来源、验证证据底线）
- `.cursor/rules/bdd.mdc` 与 `.cursor/rules/api-testing.mdc`（BDD、完整消息名、`qa/api/{domain}.yaml`）
- 涉及 proto 生成：`.cursor/skills/proto-codegen-tool/SKILL.md`
- 涉及 Handler 生成：`.cursor/skills/handler-generator-tool/SKILL.md`
- 涉及集成测试：`.cursor/skills/bdd-qa/SKILL.md`

## 执行流水线（禁止跳步；禁止无证据宣称完成）

### 1. 任务分解

将需求拆成可验证子任务；标明依赖的 **领域** 与 **qa/api 中的 domain**（若尚不存在 yaml，在阶段0 一并规划）。

### 2. 阶段0：PB 与客户端对齐（未完成不得改 `srv/` 大块业务）

- 明确：消息全名、字段、错误语义、兼容策略（新增字段/枚举）。
- 若需求中缺少与客户端/策划的约定：**列出具体问题**，停在此处等用户答复；**不要猜测**协议字段。
- 对齐结论用简短小节写进对话（或计划文档），再进入实现。

### 3. 生成链

按仓库规范执行（参数以各 Skill 为准），例如：

- 修改/新增 `proto/` 后：使用项目规定的 proto 生成工具（**禁止**手写绕过生成链）。
- 修改/新增 `.lp` 后：运行 Handler 生成工具，更新 `common/handlergen/` 等。
- `qa/api/{domain}.yaml` 与 PB/OpenAPI 产出保持一致，**禁止编造**接口。

### 4. BDD 先于或并行于实现（按项目 TDD/BDD 约定）

- 从 `qa/api/{domain}.yaml` 取消息名与字段；编写 `qa/integration/{领域}/` 下用例。
- 失败场景优先于成功场景（见 `bdd.mdc`）。

### 5. 实现

在 `srv/` 等目录按 DDD/项目惯例实现；日志与错误码符合规范。

### 6. 验证循环（必须有终端输出证据）

- 运行相关 **`go test`**（至少覆盖本次改动的包或 `qa/integration` 下对应领域）。
- **未完成验证、未粘贴命令输出摘要前**，不得写「已完成」「已通过」。
- 失败则修代码或测试，重复直到通过或明确环境/数据阻塞。

### 7. 收尾

- 简要说明：改了哪些文件、如何验收、若需合库可建议的 commit message 类型（遵守 `git-commit-message` 规则）。

---

## 自动化语义（对用户承诺的边界）

- **「自己做完」**：在 Agent 模式 + 权限足够时，Agent 应 **主动** 完成读写、生成、测试，**不因「下一步是否继续」而停止**；仅当阶段0 缺信息、或测试依赖外部服务/账号不可用、或用户取消时停止。
- **不能替代的部分**：PB 与客户端未对齐时的业务决定、生产密钥、你未提供的 GM/测试账号，须由你补充。

---

## 与脚本 `scripts/harness.sh` 的关系

仓库中 `scripts/harness.sh` 仅用于 **一键执行本地生成/测试命令**；本命令描述的是 **AI 编排层**。Agent 可在验证阶段调用该脚本或等价命令，以减少手敲路径。
