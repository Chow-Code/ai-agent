# 测试用例文档生成规则

> **BDD 与可执行用例的完整规范**在技能 **`bdd-qa`**（`.cursor/skills/bdd-qa/SKILL.md`）。本文件是 **Cursor 命令**：只规定「测试用例 **Markdown** 」写到哪里、有哪些章节；不写 `go test` 细节。

## 文档适用范围

**本规则文档用于生成「测试用例文档」，作为 BDD 接口测试与 TDD 开发的前置产出物。**

- **适用范围**：
  - ✅ 基于开发文档（DD-2 用户场景、DD-3 功能需求、DD-6 测试策略）与功能协议（服务器需求 SR-3、proto、qa/api）生成测试用例文档
  - ✅ 测试用例文档放在 `docs/design/server/{系统名称}/` 下，与开发文档、服务器需求同目录
  - ✅ 为后续「先写 BDD 测试代码、再实现功能」的 TDD 流程提供可执行测试条目
  - ✅ **无对外接口或响应未展示的业务**：部分业务可能没有独立对外接口，或接口返回内容中未体现该行为（如登录时内部恢复养气属性、事件回调等），可通过**服务端日志分析**进行验证，测试条目中注明「验证方式：日志」
  - ✅ **无对外接口时的触发方式**：部分业务没有对外接口时，除通过登录/其他接口间接触发外，可考虑使用 **GM 命令** 触发该业务，测试条目中注明「触发方式：GM」及命令格式（如 `,,命令名,参数1,参数2`），便于 BDD 通过 HTTP 调用 `/api/command` 或 testutil 执行

- **不适用范围**：
  - ❌ 不直接生成 BDD 代码（BDD 代码由开发实施阶段根据本测试用例文档编写）
  - ❌ 不替代开发文档中的 DD-2、DD-6 章节（测试用例文档是提炼与协议绑定的可执行条目）

## 文档目标

1. **统一测试条目**：将开发文档中的验收场景、功能需求转化为 Given-When-Then 的测试条目，并绑定协议消息名与 API 文档
2. **支持 TDD**：开发实施时先根据本文档生成 BDD 测试代码，再实现功能，用测试驱动开发
3. **防止幻觉**：测试条目必须引用 `qa/api/{domain}.yaml` 与完整消息名，禁止编造接口
4. **每协议用例数量**：对 `qa/api/{domain}.yaml` 中 `paths` 下**每个业务接口**（每个 `MsgCtrReq*` 请求协议），在 TC-3 中须规划 **≥2 条且 ≤5 条**测试条目（含失败/边界/成功场景的组合）；**不得**仅 1 条「能收到响应」的用例敷衍。若某协议因客观原因不足 2 条（如纯透传、仅健康检查），须在 TC-3 该协议小节下**显式说明原因**。
5. **数据正确性（强制）**：Then **不得**仅写「收到某响应」或「收到数据」；必须写清可断言的**业务字段**（与 `qa/api` schema 一致）：如 `Ret`、`Ret` 与错误码场景、列表长度、关键 ID/枚举值、数值与配置/前置条件一致、写操作后**全量查询对比**或 **Redis** 等（与 `.cursor/rules/bdd.mdc`、开发文档 DD-2「写操作后的全量拉取与对比」一致）。
6. **测试执行 JSON 报告**：BDD 跑通后须产出 **两份** JSON（见 `qa/test_reports/README.md`）：**汇总**（`bdd-report-summary.json`）与**详细**（`bdd-report-detail.json`），用于记录本次运行涉及的**消息收发与关键字段**，避免事后总结时编造未发生的请求/响应。
7. **覆盖非接口可观测行为**：对无对外接口或接口返回中未展示的业务（如登录时内部恢复属性、事件回调、后台逻辑），通过**日志验证型**测试条目描述，Then 约定为「通过分析 game 服务日志（tlog）中的关键信息进行断言」
8. **无接口时的触发方式**：对没有独立对外接口的业务，可在测试条目中约定**触发方式**为「接口间接触发」（如登录、某请求）或 **GM 命令**（如 `,,命令名,参数`）；GM 调用方式见 `.cursor/rules/gm.mdc`，BDD 中可用 testutil 或 HTTP `POST .../api/command` 执行

## 前置条件 *(mandatory)*

- ✅ **开发文档已存在**：`docs/design/server/{系统名称}/{系统名称}-开发文档.md`，且至少包含 DD-1（元信息）、DD-2（用户场景与验收场景）、DD-6（测试策略）；涉及接口的须有协议定义
- ✅ **协议来源明确**：服务器需求文档（SR-3 协议设计）或 `proto/`、`qa/api/{domain}.yaml` 中已有本功能相关消息定义
- ✅ **系统名称一致**：与开发文档、服务器需求文档中的「系统名称」一致，用于目录与文件名

**若开发文档尚未生成或未包含 DD-2/DD-6，须先使用 development 命令完成开发文档后再执行本命令。**

## 文档生成流程

```
步骤1：输入准备
    ↓
步骤2：生成测试用例文档 *(mandatory)*
    ↓
步骤3：文档检查与保存
    ↓
【本命令结束】开发实施时先基于本文档生成 BDD 测试代码，再按 TDD 实现功能
```

### 步骤1：输入准备

- 确认开发文档路径：`docs/design/server/{系统名称}/{系统名称}-开发文档.md`
- 确认关联的服务器需求文档路径（开发文档 DD-1 中的「关联文档」）
- 确定涉及的 **领域/子域** 与 **API 文档**：`qa/api/{domain}.yaml`（若尚无，需在测试用例文档中标注「待补充 API 文档」）
- 从开发文档中提取：DD-2 用户故事与验收场景（Given-When-Then）、DD-3 功能需求与业务模块、DD-6 测试策略中的接口测试场景
- 从服务器需求文档或 proto/qa/api 中提取：本功能涉及的请求/响应/通知消息名称（完整名，如 `MsgCtrReqXxx`、`MsgCtrResXxx`、`MsgCtrNtfXxx`）

### 步骤2：生成测试用例文档 *(mandatory)*

**保存路径**：`docs/design/server/{系统名称}/{系统名称}-测试用例.md`

**文档结构**：见下方「测试用例文档内容」章节。

**生成要求**：

- 每条测试条目必须包含：**Given-When-Then**、**请求消息名**（如适用）、**预期响应/通知消息名**、**可断言结果**（须含**数据正确性**：具体字段名、类型、与前置/配置一致的值或范围，而非仅「收到响应」）；可选：前置 GM 造数、依赖的 API 文档路径
- **每协议条目数**：对每个 `MsgCtrReq*` 业务协议，在 TC-3 中列出 **2～5 条**条目（失败、边界、成功等组合）；不足 2 条时须在该协议下**脚注说明原因**
- 协议与消息名必须与服务器需求文档或 `qa/api/{domain}.yaml` 一致，**禁止编造消息名或字段**
- 若某接口尚未在 `qa/api/{domain}.yaml` 中定义，须在文档中标注「协议已设计，待同步至 qa/api 与 proto 生成工具」

### 步骤3：文档检查与保存

- 检查测试条目是否覆盖开发文档 DD-2 的主要验收场景与 DD-6 的接口测试要求
- 检查所有消息名是否完整且与协议/API 文档一致
- 保存到 `docs/design/server/{系统名称}/{系统名称}-测试用例.md`

## 测试用例文档内容

**文档必须包含以下章节：**

### TC-1. 文档元信息

- **功能名称**：`[与开发文档一致]`
- **创建日期**：`[日期]`
- **最后更新**：`[日期]`
- **状态**：`Draft` / `In Use`
- **关联开发文档**：`docs/design/server/{系统名称}/{系统名称}-开发文档.md`
- **关联服务器需求**：`docs/design/server/{系统名称}/{系统名称}-服务器需求.md`
- **涉及领域**：`[领域名称，如 game]`
- **API 文档**：`qa/api/{domain}.yaml`（接口定义数据来源，BDD 编写时禁止编造）

### TC-2. 协议与接口清单 *(mandatory)*

从服务器需求 SR-3 或 proto/qa/api 整理本功能涉及的接口，便于 BDD 与 API 文档对齐。

| 消息名称（完整） | 类型 | 说明 | qa/api 路径 |
|-----------------|------|------|-------------|
| `MsgCtrReqXxx` | 请求 | 简要说明 | `qa/api/{domain}.yaml` |
| `MsgCtrResXxx` | 响应 | 简要说明 | 同上 |
| `MsgCtrNtfXxx` | 通知 | 推送场景说明 | 同上 |

- 若 API 文档中尚未包含某消息，在「说明」列标注「待补充至 qa/api」

### TC-3. 测试条目 *(mandatory)*

每条条目对应一个可执行的 BDD 用例（或一组紧密相关的步骤），格式如下。

#### 条目格式

对每个 **User Story / 业务模块** 拆分为若干 **测试条目**：

- **条目编号**：`TC-E-001`、`TC-E-002`…
- **来源**：开发文档 DD-2 的哪个 User Story / 验收场景，或 DD-3 的哪个业务模块
- **Given（前置条件）**：编号列出（已登录、已有角色、已通过 GM 造数等）
- **When（操作）**：发送哪条请求消息、携带何种参数（可引用 API 文档字段）
- **Then（预期）**：可断言的结果（收到哪条响应/通知、Ret=0、某字段等于某值或非空）
- **请求消息名**：完整名称，如 `MsgCtrReqOpenKnapsack`
- **预期响应/通知消息名**：完整名称，如 `MsgCtrResOpenKnapsack`、`MsgCtrNtfXxx`
- **可选**：前置 GM（如 `,,additem,itemId,count`）、依赖的 `qa/api/{domain}.yaml` 路径

**Then 必须可客观判断**：使用「Ret=0」「某字段非空」「列表包含 N 条」等，禁止仅写「成功」「正确」。

#### 示例

```markdown
#### TC-E-001：绝学激活（道具激活）

- **来源**：DD-2 User Story 1 - 绝学激活，验收场景 1
- **Given**：
  1. 网关与 game 服可用，玩家已 TCP 登录
  2. 背包中已有激活道具（可通过 GM `,,additem,{ActivateItemID},1` 造数）
- **When**：发送 `MsgCtrReqPeerlessSkillActivate`，参数绝学 ID=10001
- **Then**：收到 `MsgCtrResPeerlessSkillActivate`，Ret=0；绝学实例创建（研习等级=1、突破等级=1）
- **请求消息名**：`MsgCtrReqPeerlessSkillActivate`
- **预期响应消息名**：`MsgCtrResPeerlessSkillActivate`
- **API 文档**：`qa/api/skill.yaml`（或对应 domain）
```

#### 边界与错误场景

- 为「重复激活」「参数非法」「配置不存在」等设计独立条目，明确 **When** 与 **Then**（如 Ret= 具体错误码）。

#### 无对外接口或响应未展示时的验证（日志分析）

**适用场景**：业务逻辑存在但满足以下任一情况时，可采用**日志验证**补充测试条目：

- 该业务**没有独立对外接口**（如登录流程中内部执行的「养气属性恢复」、事件回调触发的逻辑）
- 该业务由某接口间接触发，但**接口响应/推送中未体现该行为**（如某字段未在协议中返回，仅服务端内部写库或打日志）

**触发方式**（When 的补充约定）：

- **接口间接触发**：通过登录、发送某业务请求等间接触发（如「执行登录」触发 OnPlayerLogin）。
- **GM 命令触发**：部分业务没有对外接口时，可考虑**通过 GM 命令直接触发**。GM 调用方式：`POST {{gameServerConsoleAddr}}/api/command`，请求头 `Session-Token`（需先 Web 登录获取），请求体 `{"Command": ",,命令名,参数1,参数2,..."}`（命令以 `,,` 开头，参数逗号分隔）。可用 GM 列表见 `srv/game/handler/gm_command/`、`.cursor/rules/gm.mdc`。若该业务已提供或计划提供专用 GM 命令，在测试条目中注明「触发方式：GM」及具体命令（如 `,,xxx,arg1,arg2`）；BDD 中可用 testutil.GetSessionTokenOptional + testutil.GMCommandOptional(sessionToken, ",,命令,参数") 或直接 HTTP 调用。

**条目格式**（在 TC-3 中可单独列出或与接口型条目混排）：

- **条目编号**：`TC-E-xxx`（可与接口型统一编号）
- **来源**：开发文档 DD-2/DD-3 中对应业务模块或验收场景
- **验证方式**：**日志**（或接口响应，若 GM 返回 Result 可断言则 Then 也可基于 GM 响应）
- **触发方式**（可选）：**接口间接触发**（如登录、某请求）或 **GM 命令**（注明 `,,命令名,参数`）
- **Given（前置条件）**：与接口型相同（已登录、造数等）；若由 GM 触发，需先获取 Session（Web 登录）
- **When（操作）**：执行会触发该业务的操作（如登录、发送某请求、**调用 GM 命令** `,,命令名,参数`）
- **Then（预期）**：**通过分析 game 服务日志**可得出可断言结果，例如：
  - 日志中在对应请求/会话后出现**指定关键字**（如业务模块名、关键参数、tlog 消息）
  - 日志**级别**与内容符合预期（如 Info 中包含某成功标识、Error 未出现）
  - 日志**链路**完整（如 Handler → AppService → Domain 的 tlog 按序出现）
  - 若 When 为 GM 命令，则还可断言 GM 响应中 `Ret=0`、`Result` 含预期内容
- **日志位置**：`logs/`（项目根或部署目录下，各服务按实例分文件）；**业务主日志在 game 服务**，参考 `.cursor/rules/testing.mdc` 的「日志分析规则」
- **可选**：注明期望的 tlog 关键字、日志级别、或「人工/脚本在用例执行后抓取日志做断言」

**Then 仍须可客观判断**：例如「game 日志中在本次登录后出现包含 'PeerlessSkill' 与 'ApplyAllRoles' 的 Info 行」，禁止仅写「日志正常」。

### TC-4. BDD 与 TDD 使用说明

- **BDD 代码位置**：`qa/integration/{领域}/`，参考 `.cursor/rules/testing.mdc`、`.cursor/rules/testing.mdc`
- **数据来源**：请求/响应字段、消息名称以 `qa/api/{domain}.yaml` 为准，禁止在测试中编造
- **TDD 流程**：开发实施时先根据本文档的测试条目编写 BDD 测试代码（Ginkgo + Gomega），再实现 Handler/AppService/Domain，使测试通过（红→绿→重构）
- **执行命令**：`go test -v ./qa/integration/{domain}/` 或 `go test -v ./qa/integration/...`
- **测试完成后的 JSON 报告（防幻觉）**：用例通过或用于汇报/评审时，须在 `qa/test_reports/` 下生成 **`bdd-report-summary.json`**（汇总）与 **`bdd-report-detail.json`**（逐条用例的消息收发与断言要点），格式与字段见 **`qa/test_reports/README.md`**；不得仅凭记忆口述「测了哪些协议」而不落盘对照。
- **日志验证**：当测试条目标注「验证方式：日志」时，断言依赖** game 服务日志**（`logs/` 下对应实例、tlog 输出）。BDD 可实现为：用例执行完请求/操作后，读取或拉取 game 日志，对日志内容做关键字/正则断言；或先执行用例，再人工/脚本根据文档中的 Then 检查日志。**玩家业务主日志在 game 服务**，请求链路为 gateway → waypoint → game，排查时以 game 日志为准；详见 `.cursor/rules/testing.mdc` 的「日志分析规则」。
- **GM 命令触发**：当测试条目标注「触发方式：GM」时，When 通过 **GM 命令** 触发业务。BDD 中先通过 Web 登录获取 Session（如 testutil.GetSessionTokenOptional），再调用 `POST {{gameServerConsoleAddr}}/api/command` 或 testutil.GMCommandOptional(sessionToken, ",,命令名,参数1,参数2") 执行；GM 命令格式与可用命令见 `.cursor/rules/gm.mdc`、`srv/game/handler/gm_command/`。

## AI 使用指南

当 AI 使用本规则生成测试用例文档时：

1. **必须读取**：对应系统名称下的开发文档（至少 DD-1、DD-2、DD-6）与服务器需求文档（SR-3 协议）
2. **必须对齐**：每条测试条目的消息名与协议/API 文档一致；若 qa/api 中暂无，在文档中标注
3. **必须可执行**：Given-When-Then 与 Then 的可断言性满足 testing.mdc 要求
4. **保存路径**：`docs/design/server/{系统名称}/{系统名称}-测试用例.md`
5. **不生成 BDD 代码**：本命令只产出测试用例文档，BDD 代码在开发实施阶段由 development 流程中的 TDD 步骤完成
6. **无接口/响应未展示时**：若开发文档或业务细节中存在「无独立对外接口」或「接口返回中未展示」的逻辑（如登录时内部恢复属性、事件回调），须在 TC-3 中增加**日志验证型**条目，注明「验证方式：日志」、When 的触发操作、Then 的日志可断言内容（关键字、级别、链路）及日志位置（game 服务、logs/），便于后续通过日志分析验证
7. **无接口时的触发方式**：对没有独立对外接口的业务，在条目中可约定 **触发方式** 为「接口间接触发」或 **「GM 命令」**。若该业务可由 GM 命令触发（现有或计划新增的 GM），须注明「触发方式：GM」及具体命令格式（如 `,,命令名,参数`），参考 `.cursor/rules/gm.mdc`、`srv/game/handler/gm_command/`；BDD 中通过 Session + `/api/command` 或 testutil.GMCommandOptional 执行
8. **每协议 2～5 条与数据断言**：生成 TC-3 时核对每个 `MsgCtrReq*` 是否已有 2～5 条条目且 Then 含具体字段断言；**测试执行后**补充或核对 `qa/test_reports/` 下两份 JSON 是否与真实收发一致

## 与 development 命令的关系

- **test-case 命令**：基于开发文档与协议，产出「测试用例文档」到 `docs/design/server/{系统名称}/`
- **development 命令**：在步骤 6 开发实施中，先基于测试用例文档生成 BDD 测试代码，再按实施计划实现功能（TDD）
- 测试用例文档可在开发文档分步完成后生成（建议在 DD-2、DD-6 及协议稳定后生成），也可在开发实施前一次性生成
