# 缘分技能升级条件检查（迭代：多宠取最低阶）

## 与策划/需求对齐（2026-03-30 迭代）

用户明确：**缘分涉及几只宠物时，取各宠中的「最低阶」** 作为共同约束依据（短板）。

### 须区分的两段逻辑

1. **缘分激活 / 升级（FateUpgrade，P-16）**
  - 配置仍为 `**PetGroup` + `PetChange` 等长**：第 `i` 只宠物须 **各自** 满足 `PetChange[i]` 对应的品阶/「变」要求（**逐宠校验**，不能用「只比 PetGroup 里最低阶」替代整表语义）。  
  - 实现 FateUpgrade 时仍以设计文档 **逐条 PetId + 品阶** 为准。
2. **缘分技能等级上限（SkillLevelUp 分支）**
  - 本条为 **多宠共同上限**：对 **已激活** 的该条缘分，在 `PetGroup` 内对每个已激活宠物取当前升阶可比标量（默认 `**MajorGrade`**），再取 `**min`**，得到 **cap**。  
  - **同一 `skillId` 出现在多条已激活缘分** 时：对每条缘分算一个 `cap_f = min(PetGroup)`，再 `**min(cap_f)`**（最严）。  
  - 与用户表述 **「涉及到几个宠物，取最低的阶」** 一致。

### 实现侧不变更的要点

- 仍新增错误码 `**PetFateSkillEvolveCap`**（或已定名），在 `newLevel > cap` 时返回。  
- 仍要求技能出现在 `CtPetFate.FateSkill` 且至少一条对应缘分 `**Codex.Fates[fateId].Level >= 1`**，否则走 `**PetSkillNotActivated`**。  
- **MajorGrade 与技能等级 1:1** 仍为默认假设，注释中保留「若策划以 PetChange 刻度为准则再映射」。

## 原实施顺序（摘要）

1. `EMsgError.proto` + proto 生成
2. `PetService.SkillLevelUp` 接入缘分技能 cap（**min 阶** 规则如上）
3. `qa/api/pet.yaml` 等补错误码
4. 测试 + `srv/game/domain/pet/README.md` + 操作记录
